Tổng Quan CNTT Hiện Đại

Lịch sử phát triển hạ tầng: Physical → VM → Cloud → Cloud Native, DevOps và IaC

Nguồn: CNCF, AWS, Microsoft Azure

Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu lịch sử phát triển hạ tầng CNTT từ physical server đến cloud native. Phân biệt được các mô hình triển khai. Nắm được khái niệm DevOps, CI/CD và Infrastructure as Code — nền tảng cho toàn bộ giáo trình.

Physical Server (Bare Metal)

Khái niệm

Physical server là máy chủ vật lý — phần cứng thực (CPU, RAM, disk, network card) đặt trong data center. Mỗi ứng dụng chạy trực tiếp trên hệ điều hành của máy chủ.

Đặc điểm

  • 1 server = 1 ứng dụng (thường) — tránh conflict giữa các app
  • Provisioning chậm — mua hardware, ship, rack, cài OS mất tuần/tháng
  • Utilization thấp — trung bình chỉ 10-15% CPU được sử dụng
  • Scaling = mua thêm máy — không thể scale nhanh
  • Chi phí cao — CapEx lớn, cần đội vận hành phần cứng

Case Study ngân hàng

Trước 2010, hầu hết ngân hàng Việt Nam chạy core banking trên IBM Power Systems hoặc Oracle Exadata — bare metal, single point of failure, HA bằng clustering hardware đắt đỏ.

Virtual Machine (Ảo hóa)

Khái niệm

Hypervisor (VMware ESXi, Microsoft Hyper-V, KVM) tạo nhiều máy ảo (VM) trên cùng 1 physical server. Mỗi VM có OS riêng, tài nguyên riêng, cách ly hoàn toàn.

Bước nhảy quan trọng

Physical → Hypervisor → VM 1 + VM 2 + VM n

Ưu điểm so với Physical

  • Utilization tăng lên 60-80% (nhiều VM chia sẻ 1 host)
  • Provisioning nhanh hơn — tạo VM trong phút, không phải tuần
  • Isolation tốt — mỗi VM có OS riêng, security boundary rõ ràng
  • Snapshot/Clone — backup và nhân bản dễ dàng
  • Live Migration — di chuyển VM giữa hosts không downtime

Hạn chế

  • Mỗi VM chạy full OS → tốn RAM, disk (2-4GB OS overhead/VM)
  • Boot time: 30 giây - vài phút
  • Density bị giới hạn bởi overhead OS

Cloud Computing

Khái niệm

Cloud Computing cung cấp tài nguyên CNTT (compute, storage, network) on-demand qua internet, trả phí theo sử dụng (pay-as-you-go).

5 đặc điểm theo NIST

  1. On-demand self-service — tự provision không cần human intervention
  2. Broad network access — truy cập qua mạng từ mọi thiết bị
  3. Resource pooling — tài nguyên pooled, phục vụ nhiều tenant
  4. Rapid elasticity — scale up/down nhanh chóng
  5. Measured service — đo lường và tính phí theo usage

3 mô hình dịch vụ

Mô hìnhBạn quản lýProvider quản lýVí dụ
IaaSOS, Runtime, App, DataHardware, Network, StorageAWS EC2, Azure VM
PaaSApp, DataOS, Runtime, MiddlewareHeroku, Azure App Service
SaaSCấu hình, DataToàn bộ stackGmail, Salesforce

4 mô hình triển khai

  • Public Cloud — AWS, Azure, GCP — shared infrastructure, pay-per-use
  • Private Cloud — OpenStack, VMware vSphere — dedicated, kiểm soát hoàn toàn
  • Hybrid Cloud — kết hợp public + private — workload sensitive on-prem, còn lại trên cloud
  • Multi-Cloud — sử dụng nhiều public cloud — tránh vendor lock-in
Case Study: Ngân hàng và Cloud Ngân hàng Việt Nam thường dùng Private Cloud hoặc Hybrid Cloud do yêu cầu compliance (NHNN, Thông tư 09). Dữ liệu khách hàng core phải on-prem, nhưng non-sensitive workloads (CRM analytics, marketing) có thể lên public cloud.

Cloud Native

Khái niệm

Cloud Native là cách thiết kế, xây dựng và vận hành ứng dụng tận dụng tối đa mô hình cloud computing. Theo CNCF:

"Cloud native technologies empower organizations to build and run scalable applications in modern, dynamic environments such as public, private, and hybrid clouds."

DevOps

Khái niệm

DevOps là văn hóa và tập hợp practices kết hợp phát triển phần mềm (Dev) và vận hành IT (Ops) để rút ngắn development lifecycle và delivery liên tục với chất lượng cao.

Vòng đời DevOps (∞ loop)

Plan → Code → Build → Test → Release → Deploy → Operate → Monitor

Nguyên tắc CALMS

  • Culture — Phá bỏ silos giữa Dev và Ops, shared responsibility
  • Automation — Tự động hóa mọi thứ có thể (build, test, deploy, monitoring)
  • Lean — Loại bỏ waste, tối ưu flow, batch nhỏ
  • Measurement — Đo lường mọi thứ (DORA metrics, SLOs)
  • Sharing — Chia sẻ kiến thức, tools, trách nhiệm

CI/CD

Continuous Integration (CI)

Developers merge code vào shared branch thường xuyên (nhiều lần/ngày). Mỗi merge trigger automated build và test.

  • Phát hiện conflicts và bugs sớm
  • Automated build + unit tests + static analysis
  • Feedback trong phút, không phải ngày
  • Artifact: Docker image, JAR, WAR

Continuous Delivery (CD)

Code luôn ở trạng thái sẵn sàng deploy production. Mỗi change đi qua pipeline tự động, nhưng deploy production cần approval.

Continuous Deployment

Tự động deploy production không cần approval — mỗi commit pass tests sẽ tự deploy. Yêu cầu mature testing và monitoring.

Commit → Build → Unit Test → Integration Test → Staging → Production

Infrastructure as Code (IaC)

Khái niệm

Quản lý và provision infrastructure bằng code thay vì manual processes. Infrastructure được khai báo trong files, version-controlled, và reproducible.

2 approaches

DeclarativeImperative
Khai báo trạng thái mong muốnKhai báo từng bước thực hiện
Terraform, Kubernetes YAML, CloudFormationAnsible, scripts
"Tôi muốn 3 VMs với 4GB RAM""Tạo VM 1, set RAM 4GB, tạo VM 2..."
Idempotent — chạy nhiều lần kết quả giống nhauPhải xử lý state manually

Lợi ích IaC

  • Reproducible — tạo lại environment identically từ code
  • Version controlled — track changes, rollback, audit trail
  • Consistent — dev/staging/production giống nhau (no more "works on my machine")
  • Testable — validate infrastructure changes trước khi apply
  • Scalable — provision 100 servers dễ như 1 server

Công cụ phổ biến

  • Terraform — multi-cloud IaC, declarative HCL language
  • Ansible — configuration management, agentless
  • Helm — Kubernetes package manager
  • Pulumi — IaC bằng programming languages (TypeScript, Python, Go)

Tóm Tắt Chương

Timeline phát triển Physical (1960s-2000s) → Virtualization (2000s) → Cloud (2006+) → Cloud Native (2013+). Mỗi bước giảm overhead, tăng tốc provisioning, và cho phép scale linh hoạt hơn.
