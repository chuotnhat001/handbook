Kafka & Event-Driven Architecture

Broker, Topic, Partition, Consumer Group, Saga Pattern, Outbox Pattern, Dead Letter Queue

Nguồn: Apache Kafka Docs, Confluent

Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu tại sao cần message queue trong microservices. Nắm kiến trúc Kafka (broker, topic, partition, consumer group). Biết các event-driven patterns quan trọng: Saga, Outbox, Dead Letter Queue. Áp dụng vào case study ngân hàng thực tế.

Tại Sao Cần Message Queue

Vấn đề với synchronous communication

  • Tight coupling — Service A gọi trực tiếp Service B → A phụ thuộc B available
  • Cascading failures — B down → A timeout → caller of A timeout → domino effect
  • Scalability bottleneck — B chậm → A bị block → toàn bộ chain chậm
  • Temporal coupling — cả hai phải online cùng lúc

Message Queue giải quyết

  • Decoupling — Producer và Consumer không biết nhau, chỉ biết queue
  • Resilience — Consumer down → messages buffer trong queue → process khi up lại
  • Scalability — thêm consumers để parallelize processing
  • Peak handling — buffer burst traffic, process at consumer's pace

Kafka Architecture

Apache Kafka là distributed event streaming platform — distributed commit log, ordered, immutable, durable.

Producers → Kafka Cluster (Brokers) → Consumers

Topic & Partition

Topic

Logical channel cho messages cùng loại. Ví dụ: crm.customer.created, payment.transaction.completed.

Partition — Key to Scalability

  • Mỗi topic có N partitions (configure khi tạo topic)
  • Ordering guarantee chỉ trong 1 partition (không cross-partition)
  • Parallelism — N partitions = max N consumers parallel
  • Partition key — quyết định message đi vào partition nào (hash(key) % N)

Replication

  • Mỗi partition có replicas trên nhiều brokers
  • Leader — handle reads/writes cho partition
  • Follower — replicate data từ leader, take over nếu leader fail
  • Replication factor = 3 → tolerate 2 broker failures
Chọn partition key cẩn thận Messages cùng key luôn vào cùng partition → guaranteed order. Ví dụ ngân hàng: dùng customer_id làm key → mọi events của 1 khách hàng đều ordered. Sai key → mất ordering guarantee.

Producer

Idempotent Producer

Enable enable.idempotence=true → Kafka đảm bảo mỗi message ghi đúng 1 lần dù producer retry. Quan trọng cho financial transactions.

Consumer & Consumer Group

Consumer Group

Nhiều consumers cùng group ID phối hợp xử lý topic:

  • Mỗi partition assigned cho đúng 1 consumer trong group
  • Thêm consumers (lên tới số partitions) → tăng throughput
  • Consumers > partitions → some consumers idle
  • Consumer fail → rebalance, partitions reassign cho consumers còn lại

Offset Management

  • Offset — vị trí đọc của consumer trong partition (sequential number)
  • Commit offset — consumer confirm đã xử lý xong đến offset N
  • Auto commit — commit định kỳ (risk: message loss nếu crash giữa chừng)
  • Manual commit — commit sau khi xử lý xong (safer, exactly-once semantics)

Event-Driven Architecture

3 Patterns chính

PatternMô tảUse case

CQRS — Command Query Responsibility Segregation

Tách write model (commands) và read model (queries). Events bridge giữa hai bên:

Command → Write DB → Event → Read DB (materialized view) → Query

Saga Pattern

Thay thế distributed transactions (2PC) trong microservices. Saga = chuỗi local transactions, mỗi step publish event trigger step tiếp theo.

2 approaches

ChoreographyOrchestration
Mỗi service listen events và reactCentral orchestrator điều phối steps
Decentralized, loose couplingCentralized logic, dễ hiểu flow
Khó trace, debug phức tạpOrchestrator = potential bottleneck
Tốt cho simple flows (3-4 steps)Tốt cho complex flows (5+ steps)

Compensating Transactions

Khi 1 step fail → rollback các steps trước bằng compensating actions:

Create Order → Reserve Stock → Charge Payment ✗ → Release Stock → Cancel Order

Transactional Outbox Pattern

Vấn đề

Service cần vừa update database VÀ publish event. Nếu DB commit nhưng event publish fail → inconsistency.

Giải pháp

  1. Write business data + event vào cùng 1 DB transaction (event vào outbox table)
  2. Separate process (CDC hoặc poller) đọc outbox table → publish lên Kafka
  3. Sau khi publish thành công → mark event as sent
-- Trong cùng 1 transaction
BEGIN;
  INSERT INTO customers (id, name, email) VALUES (...);
  INSERT INTO outbox (event_type, payload, created_at)
    VALUES ('CustomerCreated', '{"id":...}', NOW());
COMMIT;
-- Debezium CDC capture outbox changes → publish to Kafka
Debezium Debezium là CDC (Change Data Capture) platform — capture database changes và stream lên Kafka. Kết hợp với Outbox pattern = guaranteed event publishing không cần dual-write.

Dead Letter Queue (DLQ)

Khái niệm

Messages không thể xử lý (poison messages) sau N retries → chuyển sang DLQ thay vì block queue hoặc drop message.

Strategy

  1. Retry — thử lại N lần với exponential backoff
  2. DLQ — sau N retries, move đến dead letter topic
  3. Alert — notify team có messages trong DLQ
  4. Manual review — team investigate và fix hoặc replay

Khi nào message vào DLQ

  • Deserialization failure (message format invalid)
  • Business validation failure (thiếu required field)
  • Downstream dependency permanently unavailable
  • Processing timeout exceeded

Case Study: Event-Driven Banking

Scenario: Khách hàng chuyển tiền

Transfer Request → Payment Service → Kafka Event

Consumers (parallel, independent)

Consumer ServiceXử lý
Account ServiceCập nhật số dư tài khoản
CRM ServiceUpdate transaction history, customer profile
Loyalty ServiceTính điểm thưởng cho giao dịch
Fraud DetectionPhân tích realtime: unusual pattern?
Notification ServiceGửi SMS/push notification cho khách
Reporting ServiceAggregate vào data warehouse

Lợi ích

  • Payment Service không biết/care các consumers — hoàn toàn decoupled
  • Thêm consumer mới (ví dụ: Tax Reporting) → zero code change ở Payment
  • Mỗi consumer xử lý theo tốc độ riêng — không block nhau
  • Fraud Detection down 5 phút → messages buffer → process catch-up khi up lại
  • Audit trail compliant — events immutable, replay bất cứ lúc nào
Lỗi thường gặp 1) Không set partition key → mất ordering per customer. 2) Auto-commit offset → message loss khi crash. 3) Không có DLQ → poison message block consumer forever. 4) Topic naming không convention → chaos khi 50+ topics.
