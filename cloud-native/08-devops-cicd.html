Git & Branching Strategies

Git Flow

BranchVai tròLifetime
mainProduction code, always deployablePermanent
developIntegration branch cho next releasePermanent
feature/*New features, branch từ developTemporary
release/*Prepare release, bug fixes onlyTemporary
hotfix/*Production fixes, branch từ mainTemporary

Trunk-Based Development (phổ biến hơn cho CI/CD)

  • Mọi người commit vào main (hoặc short-lived branches merge trong <1 ngày)
  • Feature flags thay feature branches cho long-lived work
  • Phù hợp với continuous deployment — mỗi commit = potential release
  • Yêu cầu: strong CI, automated testing, feature flags
Ngân hàng thường dùng Git Flow hoặc biến thể (với thêm branch cho environments): develop → staging → UAT → production. Lý do: compliance yêu cầu traceability, approval gates, audit trail cho mỗi release.

Build Pipeline

CI Pipeline (Continuous Integration)

Checkout → Build → Unit Test → SAST Scan → Build Image → Image Scan → Push Registry

CD Pipeline (Continuous Delivery)

Deploy Dev → Integration Test → Deploy Staging → UAT → Approval → Deploy Prod

Image Tagging Strategy

  • Semantic versioning: crm-api:1.2.3 — cho releases
  • Git SHA: crm-api:abc1234 — cho traceability
  • Combined: crm-api:1.2.3-abc1234 — best of both
  • Tránh: latest — không biết version nào đang chạy

Helm — Kubernetes Package Manager

Khái niệm

Helm packages Kubernetes manifests thành reusable charts. Tương tự apt/yum cho Linux, nhưng cho K8s resources.

Cấu trúc Helm Chart

crm-api/
├── Chart.yaml          # Metadata (name, version)
├── values.yaml         # Default configuration
├── values-dev.yaml     # Override cho dev environment
├── values-prod.yaml    # Override cho production
└── templates/
    ├── deployment.yaml # K8s Deployment template
    ├── service.yaml    # K8s Service template
    ├── ingress.yaml    # Ingress rules
    ├── configmap.yaml  # ConfigMap
    └── hpa.yaml        # Autoscaler

Lợi ích

  • Templating — 1 chart, nhiều environments (dev/staging/prod via values files)
  • Versioning — chart versions track cùng app versions
  • Dependency management — chart phụ thuộc charts khác
  • Rollback — helm rollback quay lại revision trước
  • Ecosystem — Bitnami, Artifact Hub cung cấp charts cho common apps

GitOps & ArgoCD

GitOps Principles

GitOps: Git là single source of truth cho cả application code VÀ infrastructure/deployment configuration. Mọi thay đổi production đều qua Git commit.

Push-based vs. Pull-based

Push-based (Traditional CI/CD)Pull-based (GitOps)
CI pipeline push changes đến clusterAgent trong cluster pull changes từ Git
CI cần cluster credentialsChỉ agent cần access, không expose ra ngoài
Jenkins, GitLab CI push kubectlArgoCD, Flux watch Git repo
Drift có thể xảy ra undetectedContinuous reconciliation — drift tự heal

ArgoCD

CNCF graduated project — GitOps continuous delivery tool cho Kubernetes.

ArgoCD Workflow

Git Repo (manifests) → ArgoCD detects diff → Sync to cluster → Health check

Key Features

  • Auto-sync — tự deploy khi Git changes (hoặc manual sync)
  • Self-healing — ai đó kubectl edit trực tiếp → ArgoCD revert về Git state
  • Multi-cluster — quản lý nhiều clusters từ 1 ArgoCD instance
  • RBAC — ai sync app nào, environment nào
  • Rollback — sync về bất kỳ Git commit nào trước đó
  • Diff visualization — UI hiển thị rõ desired vs. actual state
GitOps trong ngân hàng Git = audit trail tự nhiên. Mỗi deployment = Git commit với author, reviewer, timestamp. Compliance team có thể trace bất kỳ change nào đến người thực hiện và người approve. ArgoCD RBAC đảm bảo chỉ authorized people sync production.

Deployment Strategies

Rolling Update (Default K8s)

  • Replace pods dần dần — old pods removed khi new pods ready
  • Zero downtime, gradual transition
  • Không cần infrastructure thêm
  • Rollback: kubectl rollout undo

Blue/Green Deployment

AspectMô tả
Cách hoạt động2 environments identities (Blue = current, Green = new). Switch traffic instantaneously.
ƯuInstant rollback (switch back), full testing trên Green trước switch
Nhược2x resources (cả Blue và Green running đồng thời)
Dùng khiCần instant rollback, database migration không backward-compatible

Canary Deployment

AspectMô tả
Cách hoạt độngRoute small % traffic (5-10%) đến new version. Monitor. Dần tăng lên 100%.
ƯuDetect issues với real traffic, limit blast radius, data-driven decisions
NhượcCần sophisticated routing, monitoring phải tốt để detect issues
Dùng khiHigh-traffic services, cần validate với real users trước full rollout

So sánh

StrategyDowntimeRollbackResourceRisk
RollingZeroSlow (redeploy)1x + surgeMedium
Blue/GreenZeroInstant (switch)2xLow
CanaryZeroFast (route back)1x + canaryLowest

CI Tools: Jenkins vs. GitLab CI

AspectJenkinsGitLab CI
TypeStandalone CI serverIntegrated trong GitLab
ConfigurationJenkinsfile (Groovy DSL).gitlab-ci.yml (YAML)
Plugins1800+ plugins, ecosystem khổng lồBuilt-in features, ít cần plugins
ScalingAgents (static hoặc dynamic)Runners (easily scalable trên K8s)
Learning curveCao (Groovy, plugin conflicts)Thấp hơn (YAML declarative)
MaintenanceCao (updates, plugins, security)Thấp (SaaS) hoặc medium (self-hosted)
Phù hợpLegacy, complex pipelinesModern teams, GitOps-native
Trend 2024-2025 Xu hướng chuyển từ Jenkins sang GitLab CI/Tekton + ArgoCD. Lý do: Jenkins maintenance overhead cao, plugin conflicts, security issues. GitOps model (ArgoCD) tách CI (build) khỏi CD (deploy) — mỗi phần dùng tool tốt nhất.

Tóm Tắt

Best Practices

  • Tách CI (build + test + scan + push image) khỏi CD (deploy to environments)
  • Git repo riêng cho app code và deployment config (GitOps best practice)
  • Mọi environment diff chỉ ở values file, không ở templates
  • Image immutable — same image qua dev → staging → prod, chỉ config khác
  • Automated rollback khi health check fail
