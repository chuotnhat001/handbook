OKD / OpenShift

Architecture, Route, ImageStream, BuildConfig, Operator, SCC, Multi-tenancy, CI/CD tích hợp

Nguồn: OKD Documentation, Red Hat OpenShift Docs

Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu OKD/OpenShift là gì và khác Kubernetes thuần như thế nào. Nắm các concepts riêng: Route, ImageStream, BuildConfig, Operator, SCC. Biết cách tổ chức multi-tenancy với Projects và CI/CD tích hợp sẵn.

OKD vs. OpenShift vs. Kubernetes

OKD là upstream community distribution của Red Hat OpenShift — tương tự Fedora với RHEL. OpenShift = OKD + enterprise support + certified operators.

AspectKubernetesOKD / OpenShift
BaseContainer orchestration engineKubernetes + platform features
OSAny LinuxFedora CoreOS (immutable, auto-update)
Runtimecontainerd / CRI-OCRI-O (only)
NetworkingIngress ControllerRouter (HAProxy) + Route objects
RegistryExternal (Harbor, ECR)Internal Registry built-in
BuildExternal CI (Jenkins, GitLab)BuildConfig + Source-to-Image (S2I)
SecurityPodSecurityStandardsSCC (Security Context Constraints)
Multi-tenancyNamespace + RBACProject (= namespace + extra policies)
ConsoleDashboard (basic)Web Console (full-featured)
Tại sao ngân hàng chọn OKD/OpenShift Enterprise support, hardened security (SCC), integrated CI/CD, compliance certifications, multi-tenancy native, operator ecosystem cho databases và middleware. Phù hợp regulated industries.

OKD Architecture

Layers

Fedora CoreOS → CRI-O → Kubernetes → OKD Platform Services

Platform Services (thêm so với K8s thuần)

  • Router — HAProxy-based ingress, TLS termination
  • Internal Registry — image storage trong cluster
  • OAuth Server — authentication tích hợp (LDAP, AD, OIDC)
  • Web Console — UI quản lý đầy đủ
  • Monitoring Stack — Prometheus + Grafana built-in
  • Logging Stack — EFK (Elasticsearch, Fluentd, Kibana) hoặc Loki
  • Operator Lifecycle Manager (OLM) — quản lý operators

Project & Multi-tenancy

Project = Namespace + Policies

Project là Kubernetes namespace với thêm:

  • Resource Quotas — giới hạn CPU, RAM, pods, PVCs per project
  • Limit Ranges — default requests/limits cho containers
  • Network Policies — isolate traffic giữa projects
  • RBAC — ai được làm gì trong project
  • Self-provisioning — developers tự tạo project (với quota)

Tổ chức multi-tenancy ngân hàng

# Ví dụ tổ chức projects
crm-dev          # CRM team - development
crm-staging      # CRM team - staging
crm-prod         # CRM team - production
loyalty-dev      # Loyalty team - development
loyalty-prod     # Loyalty team - production
shared-infra     # Shared: Kafka, Redis, monitoring

Route — HTTP Routing

Route expose Services ra bên ngoài cluster qua hostname. Tương đương Ingress nhưng feature-rich hơn.

Route vs. Ingress

FeatureRouteIngress
TLS terminationEdge, Passthrough, Re-encryptDepends on controller
Weighted routingNative (A/B testing)Annotation-dependent
Sticky sessionsBuilt-inAnnotation-dependent
Wildcard domainsSupportedController-dependent
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: crm-api
  namespace: crm-prod
spec:
  host: crm-api.apps.bank.vn
  to:
    kind: Service
    name: crm-api
    weight: 100
  tls:
    termination: edge
    insecureEdgeTerminationPolicy: Redirect
  port:
    targetPort: 8080

ImageStream

ImageStream là abstraction layer trên container images — theo dõi image tags và tự động trigger builds/deployments khi image thay đổi.

Lợi ích

  • Auto-trigger — deployment tự update khi image mới được push
  • Tag management — tag images trong cluster (latest, stable, v1.2.0)
  • Image lookup — reference images bằng tên ngắn thay vì full URL
  • Security scanning — track CVEs trên images đang dùng
  • Rollback — dễ dàng point lại tag cũ
apiVersion: image.openshift.io/v1
kind: ImageStream
metadata:
  name: crm-api
  namespace: crm-prod
spec:
  lookupPolicy:
    local: true
  tags:
  - name: latest
    from:
      kind: DockerImage
      name: harbor.bank.vn/crm/api:latest
    importPolicy:
      scheduled: true  # auto-import khi source thay đổi

BuildConfig

BuildConfig định nghĩa cách build container images ngay trong cluster. Không cần external CI cho bước build.

Source-to-Image (S2I)

Source Code + Builder Image = App Image

Developer push code → OKD tự build image → push vào internal registry → trigger deployment. Zero Dockerfile knowledge required.

Operators

Operators encode operational knowledge (cách install, configure, upgrade, backup) vào software. Kubernetes-native way để manage complex applications.

Operator Pattern

  • Custom Resource (CR) — khai báo desired state ("tôi muốn PostgreSQL cluster 3 nodes")
  • Controller — watch CR, reconcile actual state = desired state
  • Domain expertise — biết cách failover, backup, upgrade

Operators phổ biến trong ngân hàng

Operator Lifecycle Manager (OLM)

OKD quản lý operators qua OLM — install, upgrade, manage dependencies giữa operators. OperatorHub cung cấp catalog certified operators.

Security Context Constraints (SCC)

SCC kiểm soát actions mà Pod có thể thực hiện — mạnh hơn PodSecurityStandards của K8s vanilla.

Built-in SCCs (từ restrictive → permissive)

SCCCho phépUse case
restrictedNon-root, no caps, read-only rootfsDefault cho mọi workload
nonrootNon-root nhưng some capabilitiesApps cần bind ports >1024
anyuidChạy với bất kỳ UIDLegacy apps cần root
privilegedFull access (host network, devices)Infrastructure only (monitoring agents)
Security Best Practice Mặc định mọi workload chạy với SCC "restricted". Chỉ escalate khi có justification rõ ràng. Trong ngân hàng, mọi SCC exception cần security team approval.

CI/CD Tích Hợp

Built-in CI/CD Options

  • BuildConfig + Webhooks — Git push → auto build → auto deploy
  • Tekton Pipelines — Cloud-native CI/CD (CNCF project), chạy native trên K8s
  • OpenShift Pipelines (Tekton) — managed Tekton với UI
  • OpenShift GitOps (ArgoCD) — GitOps deployment operator

Typical Pipeline trên OKD

Git Push → Webhook → Tekton Pipeline → Build Image → Push Registry → ArgoCD Sync

Internal Registry Workflow

  1. Developer push code lên Git
  2. BuildConfig trigger (webhook hoặc ImageStream change)
  3. Build pod chạy, tạo image
  4. Image push vào Internal Registry
  5. ImageStream tag update
  6. Deployment detect new image → rolling update
Case Study: CRM trên OKD CRM team có 3 projects (dev/staging/prod). Code push → Tekton build trong crm-dev → test → promote image tag sang crm-staging → QA approve → ArgoCD sync sang crm-prod. Mỗi environment isolated qua Network Policies. SCC restricted, non-root containers.

Tóm Tắt

Lỗi thường gặp

  • Quên SCC → Pod fail start vì "unable to validate against any security context constraint"
  • Dùng Ingress thay vì Route → mất features (weighted routing, TLS re-encrypt)
  • Không set Resource Quotas → 1 team consume hết cluster resources
  • Build image với root user → SCC restricted block
  • Không dùng ImageStream → miss auto-rollout khi base image update security patch
