Container & Docker

Linux Namespace, cgroups, Docker Architecture, Image, Dockerfile, Network, Volume, Compose

Nguồn: Docker Documentation, Open Container Initiative

Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu container hoạt động dưới hood (namespaces + cgroups). Nắm vững kiến trúc Docker. Biết viết Dockerfile hiệu quả, quản lý networking và volumes. Sử dụng Docker Compose cho multi-container apps.

Container Là Gì

Container là một process (hoặc nhóm processes) chạy trong môi trường isolated trên cùng 1 kernel. Không phải VM — không có hypervisor, không có guest OS riêng.

Container vs. Virtual Machine

ContainerVirtual Machine
Shares host kernelOwn kernel + OS
MB footprintGB footprint
Starts in millisecondsStarts in seconds/minutes
Process-level isolationHardware-level isolation
Hàng trăm/hostHàng chục/host
Portable (build once, run anywhere)Portable nhưng nặng hơn

Linux Namespaces — Isolation

Namespaces cung cấp isolation — mỗi container "nghĩ" nó có riêng filesystem, network, PIDs.

NamespaceIsolatesÝ nghĩa
PIDProcess IDsContainer thấy PID 1 là process chính của nó
NETNetwork stackContainer có network interface riêng, IP riêng
MNTFilesystem mountsContainer có filesystem tree riêng
UTSHostnameContainer có hostname riêng
IPCInter-process communicationShared memory, message queues isolated
USERUser/Group IDsUID 0 trong container ≠ UID 0 trên host
CGROUPCgroup rootContainer không thấy cgroups ngoài scope

cgroups — Resource Limits

Control Groups (cgroups) giới hạn tài nguyên mà container được dùng. Namespaces cho isolation, cgroups cho resource control.

Resources được giới hạn

  • CPU — cpu shares, cpu quota (ví dụ: max 50% CPU)
  • Memory — memory limit (ví dụ: max 512MB RAM)
  • I/O — disk bandwidth throttling
  • PIDs — max số processes trong container
  • Network — bandwidth limits (qua tc hoặc plugins)
# Docker run với resource limits
docker run -d \
  --memory="512m" \
  --cpus="1.5" \
  --pids-limit=100 \
  my-app:latest
OOM Killer Khi container vượt memory limit, Linux OOM Killer sẽ kill container. Đây là lý do cần set memory limits hợp lý — quá thấp = random crashes, quá cao = risk cho host.

Docker Image & Layers

Khái niệm

Docker Image là ordered collection of filesystem changes + metadata. Image bao gồm nhiều layers xếp chồng, mỗi layer là một diff từ layer trước.

Union Filesystem

  • Mỗi instruction trong Dockerfile tạo 1 layer
  • Layers are read-only và shared giữa containers
  • Container thêm 1 writable layer on top (container layer)
  • Copy-on-Write: chỉ copy file khi cần modify

Layer caching

Docker cache layers. Nếu instruction không đổi → dùng cache. Thứ tự instructions ảnh hưởng build speed:

  • Đặt ít thay đổi trước (OS deps, runtime)
  • Đặt thay đổi thường xuyên cuối (source code)

Dockerfile

Cấu trúc cơ bản

# Base image
FROM eclipse-temurin:21-jre-alpine

# Metadata
LABEL maintainer="team@bank.vn"

# Install dependencies (cached layer)
RUN apk add --no-cache curl

# Copy application
COPY target/app.jar /app/app.jar

# Configuration
ENV JAVA_OPTS="-Xms256m -Xmx512m"
EXPOSE 8080

# Non-root user
RUN adduser -D appuser
USER appuser

# Entrypoint
ENTRYPOINT ["java", "-jar", "/app/app.jar"]

Multi-stage Build

Tách build stage (nặng, có compiler) và runtime stage (nhẹ, chỉ có app):

# Stage 1: Build
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /build
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: Runtime (chỉ ~100MB thay vì ~800MB)
FROM eclipse-temurin:21-jre-alpine
COPY --from=builder /build/target/app.jar /app/app.jar
USER 1000
ENTRYPOINT ["java", "-jar", "/app/app.jar"]

Docker Networking

Network Drivers

DriverUse Case
bridge (default)Containers trên cùng host communicate. Isolated network.
hostContainer dùng network stack của host trực tiếp. No isolation, max performance.
overlayMulti-host networking (Docker Swarm, Kubernetes).
noneNo networking. Complete isolation.
macvlanContainer có MAC address riêng, xuất hiện như device vật lý trên network.

DNS trong Docker

Containers trong cùng user-defined network có thể resolve nhau bằng container name — không cần hardcode IP.

# Tạo network
docker network create my-app-net

# Containers tự resolve nhau bằng name
docker run -d --name db --network my-app-net postgres:16
docker run -d --name api --network my-app-net \
  -e DB_HOST=db my-api:latest

Volumes & Storage

3 cách persist data

TypeManaged byUse case
VolumeDockerDatabase data, persistent app state
Bind MountHost filesystemDevelopment (mount source code)
tmpfsMemorySensitive data không cần persist
# Named volume (recommended cho production)
docker volume create pgdata
docker run -d --name db \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16

# Bind mount (development)
docker run -d --name api \
  -v $(pwd)/src:/app/src \
  my-api:dev

Docker Compose

Định nghĩa và chạy multi-container applications bằng 1 file YAML:

# docker-compose.yml
services:
  api:
    build: ./api
    ports:
      - "8080:8080"
    environment:
      - DB_HOST=postgres
      - REDIS_HOST=redis
    depends_on:
      postgres:
        condition: service_healthy
    networks:
      - app-net

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: crm
      POSTGRES_PASSWORD_FILE: /run/secrets/db_pass
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
    networks:
      - app-net

  redis:
    image: redis:7-alpine
    networks:
      - app-net

  pgdata:

networks:
  app-net:

Best Practices

Security

  • Non-root user — luôn chạy container với USER non-root
  • Minimal base image — Alpine, distroless, scratch — giảm attack surface
  • No secrets in image — dùng env vars hoặc secrets mount, không COPY .env
  • Scan images — Trivy, Snyk trước khi push registry
  • Pin versions — FROM node:20.11-alpine không phải FROM node:latest

Performance

  • Multi-stage builds — build image nhỏ nhất có thể
  • Layer caching — COPY package.json trước, npm install, rồi COPY source
  • .dockerignore — exclude node_modules, .git, tests
  • Combine RUN — giảm layers: RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*

Reliability

  • Health checks — HEALTHCHECK instruction cho orchestrator biết container healthy
  • Graceful shutdown — handle SIGTERM, cleanup trước khi exit
  • One process per container — không chạy app + cron + syslog trong 1 container
  • Immutable — không SSH vào container fix, rebuild image thay thế
Case Study: CRM Service CRM microservice tại ngân hàng: multi-stage build (Maven build → JRE Alpine runtime), non-root user, health check endpoint /actuator/health, graceful shutdown 30s, image scan qua Trivy trước khi push Harbor registry.
