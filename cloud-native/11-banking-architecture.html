Kiến Trúc Cloud Native Trong Ngân Hàng

CRM, BPM, Loyalty, FlowX — Toàn cảnh kiến trúc hệ thống ngân hàng trên nền tảng cloud native

Nguồn: Azure Architecture, CNCF, Red Hat OpenShift Banking

Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu toàn cảnh kiến trúc cloud native cho hệ thống ngân hàng (CRM, BPM, Loyalty). Nắm cách thiết kế microservices với bounded contexts. Biết tích hợp API Gateway, Keycloak, Kafka, Redis trong banking platform. Hiểu deployment strategy trên OKD/OpenShift với HA và DR. Có khả năng thiết kế end-to-end banking system trên cloud native stack.

Kiến Trúc CRM Microservice

Bounded Contexts

CRM ngân hàng được chia thành 4 bounded contexts chính, mỗi context là một microservice độc lập với database riêng (Database per Service pattern):

Bounded ContextResponsibilityDatabase
CustomerHồ sơ KH, KYC, segmentation, 360-viewPostgreSQL
InteractionLịch sử tương tác, tickets, notes, timelineMongoDB
CampaignMarketing campaigns, offers, targeting rulesPostgreSQL
AnalyticsCustomer scoring, churn prediction, CLVClickHouse/Elasticsearch

Communication Patterns

Customer Service → Kafka Event → Interaction Service → Analytics Service
  • Synchronous — REST/gRPC cho queries cần response ngay (lấy thông tin KH)
  • Asynchronous — Kafka events cho state changes (KH mới tạo, KH cập nhật segment)
  • CQRS — Command (write) tách riêng Query (read) cho Customer 360-view

API Gateway

Vai trò trong Banking Platform

API Gateway là single entry point cho mọi client (mobile app, internet banking, partner APIs). Trong ngân hàng, gateway đảm nhận thêm compliance và security layers.

Kong / APISIX — Lựa chọn phổ biến

FeatureKongAPISIX

PerformanceTốt (10K+ RPS/node)Rất tốt (hỗ trợ hot reload)

Gateway Responsibilities

  • Rate Limiting — 100 req/s cho mobile, 1000 req/s cho internal, 50 req/s cho partner APIs
  • Authentication — Validate JWT từ Keycloak, inject user claims vào headers
  • Routing — Path-based routing tới microservices (e.g., /api/v1/customers/* → customer-service)
  • Request/Response Transform — Mask sensitive data (PAN, SSN) trong response
  • Circuit Breaker — Protect downstream services khỏi cascade failure
  • Audit Logging — Log mọi API call cho compliance (PCI DSS requirement)

Authentication — Keycloak

OAuth2/OIDC Flow cho Banking

Mobile App → Keycloak (OIDC) → JWT + Refresh Token → API Gateway validates → Microservice

Keycloak Configuration cho Ngân Hàng

  • Realm per channel — retail-banking, corporate-banking, partner-api
  • Client per app — mobile-app, web-portal, admin-dashboard
  • Roles — customer, teller, branch-manager, admin, auditor
  • MFA required — OTP/SMS cho transactions, biometrics cho high-value
  • Token lifetime — Access: 5 min, Refresh: 30 min (banking compliance)
  • Session management — Max 1 active session per device, force logout on suspicious activity

Kafka Event Bus

Vai trò trong Banking Platform

Apache Kafka là backbone cho event-driven architecture trong ngân hàng. Mọi state change quan trọng đều publish qua Kafka, cho phép loose coupling giữa services và audit trail tự nhiên.

Topic Design cho Banking

Topic PatternVí dụMục đích
crm.customer.*crm.customer.created, crm.customer.updatedCustomer lifecycle events
payment.*payment.initiated, payment.completed, payment.failedPayment processing pipeline
loyalty.*loyalty.points.earned, loyalty.points.redeemedLoyalty program events
notification.*notification.sms, notification.push, notification.emailMulti-channel notifications
audit.*audit.login, audit.transaction, audit.config-changeCompliance audit trail

Topic Naming Convention

# Pattern: {domain}.{entity}.{event-type}
# Ví dụ banking topics:

crm.customer.created          # KH mới được tạo
crm.customer.segment-changed  # Segment thay đổi
payment.transfer.initiated    # Lệnh chuyển tiền khởi tạo
payment.transfer.completed    # Chuyển tiền thành công
payment.transfer.failed       # Chuyển tiền thất bại
loyalty.points.earned         # Tích điểm
loyalty.tier.upgraded         # Nâng hạng thành viên
audit.user.login-success      # Đăng nhập thành công
audit.user.login-failed       # Đăng nhập thất bại

Kafka Configuration cho Banking

  • Replication factor: 3 — đảm bảo durability (min.insync.replicas=2)
  • Retention: 7 ngày cho operational topics, 365 ngày cho audit topics
  • Partitions — partition by customer_id để đảm bảo ordering per customer
  • Exactly-once semantics — enable.idempotence=true cho payment topics
  • Schema Registry — Avro schemas, backward compatibility mode
  • ACLs — mỗi service chỉ produce/consume topics thuộc domain mình
Critical: Payment Event Ordering Payment events PHẢI đảm bảo ordering per transaction. Dùng transaction_id làm partition key. Nếu payment.initiated arrive sau payment.completed → data inconsistency. Enable idempotent producer + transactional consumers cho payment domain.

Caching Patterns

Cache-Aside (Lazy Loading)

# Đọc: check cache trước, miss thì query DB rồi set cache
async def get_customer(customer_id):
    # 1. Check Redis
    cached = await redis.get(f"customer:{customer_id}")
    if cached:
        return json.loads(cached)
    
    # 2. Cache miss → query PostgreSQL
    customer = await db.query("SELECT * FROM customers WHERE id = $1", customer_id)
    
    # 3. Set cache with TTL
    await redis.setex(f"customer:{customer_id}", 300, json.dumps(customer))
    return customer

Write-Through

# Ghi: update DB + cache đồng thời
async def update_customer_segment(customer_id, new_segment):
    # 1. Update PostgreSQL
    await db.execute(
        "UPDATE customers SET segment = $1 WHERE id = $2",
        new_segment, customer_id
    )
    
    # 2. Update Redis immediately
    customer = await db.query("SELECT * FROM customers WHERE id = $1", customer_id)
    await redis.setex(f"customer:{customer_id}", 300, json.dumps(customer))
    
    # 3. Publish event
    await kafka.produce("crm.customer.segment-changed", {
        "customer_id": customer_id,
        "new_segment": new_segment
    })

Redis Cluster cho Banking

  • Redis Cluster 6 nodes — 3 masters + 3 replicas (cross-AZ)
  • Persistence — AOF enabled (appendfsync everysec) cho session data
  • Memory policy — allkeys-lru cho cache, noeviction cho session/lock
  • TLS enabled — encrypt data in transit (compliance requirement)
  • AUTH + ACL — mỗi service có user riêng, chỉ access prefix thuộc mình

Database Strategy — Polyglot Persistence

Nguyên tắc: Right tool for the right job

Không một database nào phù hợp cho mọi use case. Banking platform sử dụng polyglot persistence — chọn database phù hợp nhất cho từng bounded context.

DatabaseUse CaseBounded Context
PostgreSQLOLTP, ACID transactions, relational dataCustomer, Payment, Account
MongoDBFlexible schema, document-orientedInteraction, Campaign templates
RedisCache, session, rate limitingCross-cutting (shared infra)
ElasticsearchFull-text search, log analyticsSearch, Audit logs
ClickHouseOLAP, time-series analyticsAnalytics, Reporting

PostgreSQL cho Core Banking

  • Patroni HA — automatic failover, streaming replication
  • Connection pooling — PgBouncer (transaction mode, max 200 connections/pool)
  • Partitioning — table partition by date cho transactions (tăng query performance)
  • Logical replication — replicate sang read replica cho reporting
  • Encryption — TDE (Transparent Data Encryption) cho data at rest
Database per Service Pattern Mỗi microservice sở hữu database riêng. Customer Service không được query trực tiếp vào Payment DB. Muốn lấy data cross-service → gọi API hoặc subscribe Kafka events. Điều này đảm bảo loose coupling và independent deployability.

ArgoCD GitOps

  • Git as single source of truth — mọi config change phải qua Git PR

High Availability

Multi-AZ Deployment

Banking services phải chạy across multiple Availability Zones để đảm bảo không single point of failure. Nếu 1 AZ down, traffic tự động failover sang AZ còn lại.

ComponentHA StrategyMin Replicas

Payment GatewayMulti-AZ, active-active4 (2 per AZ minimum)

KafkaMulti-AZ brokers, rack awareness3 brokers (min.insync=2)

Pod Anti-Affinity

# Đảm bảo pods của cùng service spread across các node/AZ khác nhau
apiVersion: apps/v1
kind: Deployment
metadata:
  name: crm-api
  namespace: crm-prod
spec:
  replicas: 3
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: crm-api
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values: ["crm-api"]
            topologyKey: kubernetes.io/hostname

Circuit Breaker Pattern

Closed (normal) → Open (failures threshold) → Half-Open (test) → Closed (recovered)
  • Threshold — Open circuit sau 5 consecutive failures hoặc 50% error rate trong 10s window
  • Timeout — Circuit stays open 30s trước khi thử half-open
  • Fallback — Return cached data hoặc graceful degradation message
  • Implementation — Resilience4j (Java), Istio service mesh (infra-level)

Disaster Recovery

RPO/RTO Targets

Service TierRPO (Data Loss)RTO (Recovery Time)Strategy
Tier 1: Payment0 (zero data loss)< 5 phútSynchronous replication, active-active
Tier 2: CRM, Core< 1 phút< 15 phútAsync replication, warm standby
Tier 3: Analytics< 1 giờ< 4 giờBackup restore, cold standby

Backup Strategy

  • PostgreSQL — WAL archiving liên tục + daily full backup (pg_basebackup) + PITR capability
  • MongoDB — Continuous oplog backup + daily snapshot
  • Kafka — MirrorMaker 2 replicate sang DR cluster (async)
  • Redis — AOF + RDB snapshot mỗi 15 phút → S3-compatible storage
  • etcd (K8s) — Automated snapshot mỗi giờ, retain 72h
  • Velero — Backup K8s resources (namespaces, deployments, PVCs) → offsite storage

Failover Architecture

DC1 (Primary) → Global LB detects failure → DNS failover → DC2 (DR) promoted
  • Active-Active (Payment) — Cả 2 DC handle traffic, data sync realtime
  • Active-Passive (CRM) — DC2 warm standby, promote khi DC1 fail
  • DR Drill — Test failover quarterly, document runbook, measure actual RTO
DR Testing là bắt buộc NHNN yêu cầu ngân hàng phải có DR drill ít nhất 2 lần/năm. Mọi runbook phải được test thực tế, không chỉ trên giấy. Đo actual RTO và so sánh với target. Nếu miss target → escalate và fix.

Tóm Tắt

Kiến trúc tổng thể Banking Platform

Mobile/Web → API Gateway (Kong) → Keycloak Auth → Microservices → Kafka/Redis → Polyglot DB
"Cloud native banking không chỉ là containerize legacy apps. Đó là re-think cách xây dựng hệ thống tài chính — resilient, observable, secure by design, và có khả năng scale theo demand thực tế."

Checklist triển khai Banking Platform

  • Microservices với bounded contexts rõ ràng (DDD)
  • API Gateway với rate limiting, auth, audit logging
  • Keycloak SSO với MFA cho mọi channel
  • Kafka event bus với exactly-once cho payment domain
  • Redis cluster cho caching và session management
  • Polyglot persistence — right DB for right job
  • OKD/K8s với namespace isolation và resource quotas
  • GitOps CI/CD (GitLab CI → Harbor → ArgoCD)
  • Prometheus/Grafana monitoring với SLO-based alerting
  • Multi-AZ HA với pod anti-affinity và PDB
  • DR strategy với RPO/RTO per service tier
  • Network Policies và SCC cho defense in depth

Câu hỏi ôn tập

  1. Giải thích tại sao CRM cần chia thành 4 bounded contexts thay vì 1 monolith service.
  2. So sánh cache-aside và write-through pattern. Khi nào dùng pattern nào trong banking?
  3. Tại sao payment events cần exactly-once semantics? Hậu quả nếu duplicate payment event?
  4. Thiết kế Kafka topic structure cho hệ thống Loyalty (points, tiers, rewards, promotions).
  5. RPO = 0 cho Payment service đạt được bằng cách nào? Trade-off là gì?
  6. Giải thích Network Policy isolate Payment namespace — tại sao không cho phép CRM gọi trực tiếp Payment DB?
  7. SLO 99.99% cho Payment nghĩa là error budget bao nhiêu? Khi error budget cạn kiệt, team nên làm gì?
  8. Thiết kế DR failover plan cho toàn bộ banking platform khi primary DC mất điện hoàn toàn.
