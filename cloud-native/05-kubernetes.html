Kubernetes

Cluster, Control Plane, Pod, Deployment, Service, Ingress, Storage, Autoscaling, Rolling Update

Nguồn: Kubernetes Documentation, Cluster Architecture

Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu kiến trúc Kubernetes cluster (Control Plane + Worker Nodes). Nắm vững các workload resources (Pod, Deployment, StatefulSet, DaemonSet, Job). Biết cách expose services (Service, Ingress), manage config (ConfigMap, Secret), và handle storage (PV/PVC). Hiểu autoscaling và rolling update strategies.

Kubernetes Cluster Overview

Kubernetes (K8s) là hệ thống orchestration cho containers — tự động hóa deployment, scaling, và management. Một cluster gồm:

Control Plane + Worker Node 1 + Worker Node 2 + Worker Node N

Declarative model: Bạn khai báo desired state ("tôi muốn 3 replicas của service A"), Kubernetes đảm bảo actual state = desired state liên tục.

Control Plane

Bộ não của cluster — ra quyết định global (scheduling, detecting/responding to events). Production nên chạy trên 3+ nodes cho HA.

ComponentVai trò
kube-apiserverAPI entry point. Mọi interaction (kubectl, UI, controllers) đi qua API server. Stateless, scalable horizontally.
etcdDistributed key-value store. Lưu toàn bộ cluster state. Source of truth duy nhất. Cần backup thường xuyên.
kube-schedulerQuyết định Pod chạy trên Node nào. Xét: resources, affinity, taints, topology.
kube-controller-managerChạy control loops: Node controller, ReplicaSet controller, Endpoint controller... Đảm bảo actual = desired.
cloud-controller-managerTích hợp cloud provider (load balancers, routes, volumes).

Worker Nodes

Nơi containers thực sự chạy. Mỗi node có:

ComponentVai trò
kubeletAgent trên mỗi node. Nhận PodSpec từ API server, đảm bảo containers chạy đúng spec.
kube-proxyNetwork proxy. Implement Services — rules cho traffic routing đến Pods.
Container RuntimeChạy containers (containerd, CRI-O). Kubelet giao tiếp qua CRI.

Pod — Đơn Vị Nhỏ Nhất

Pod là đơn vị deploy nhỏ nhất trong K8s. Gồm 1 hoặc nhiều containers share cùng network namespace và storage.

Đặc điểm

  • Shared network — containers trong pod communicate qua localhost
  • Shared storage — volumes mounted chung giữa containers
  • Ephemeral — pods có thể bị kill và recreate bất cứ lúc nào
  • IP riêng — mỗi pod có 1 IP trong cluster network

Pod YAML cơ bản

apiVersion: v1
kind: Pod
metadata:
  name: crm-api
  labels:
    app: crm
    tier: backend
spec:
  containers:
  - name: api
    image: harbor.bank.vn/crm/api:1.2.0
    ports:
    - containerPort: 8080
    resources:
      requests:
        memory: "256Mi"
        cpu: "250m"
      limits:
        memory: "512Mi"
        cpu: "500m"
    livenessProbe:
      httpGet:
        path: /actuator/health/liveness
        port: 8080
      initialDelaySeconds: 30
    readinessProbe:
      httpGet:
        path: /actuator/health/readiness
        port: 8080
Không tạo Pod trực tiếp Luôn dùng controller (Deployment, StatefulSet...) để quản lý Pods. Pod tạo trực tiếp sẽ không được recreate khi node fail.

Workload Controllers

Deployment (phổ biến nhất)

Quản lý stateless applications. Tạo ReplicaSet → ReplicaSet tạo Pods.

  • Declarative updates (rolling update, rollback)
  • Scale up/down dễ dàng
  • History cho rollback

ReplicaSet

Đảm bảo số lượng Pod replicas luôn = desired count. Thường không tạo trực tiếp — Deployment quản lý.

StatefulSet

Cho stateful applications (databases, Kafka, Elasticsearch). Khác Deployment:

  • Stable network identity — pod-0, pod-1, pod-2 (ordered)
  • Stable storage — mỗi pod có PVC riêng, persist qua recreate
  • Ordered deployment/scaling — tạo/xóa theo thứ tự
  • Ordered rolling updates — update từ cao xuống thấp

DaemonSet

Chạy 1 Pod trên mỗi Node (hoặc subset nodes). Use cases:

  • Log collectors (Fluentd, Fluent Bit)
  • Monitoring agents (node-exporter)
  • Network plugins (Calico, Cilium)
  • Storage daemons

Job & CronJob

  • Job — chạy 1 lần đến completion (batch processing, migrations)
  • CronJob — chạy theo schedule (backup hàng đêm, reports)
ControllerUse CaseVí dụ ngân hàng
DeploymentStateless servicesCRM API, Notification Service
StatefulSetStateful workloadsPostgreSQL, Kafka brokers
DaemonSetPer-node agentsLog collector, monitoring
JobOne-time tasksDB migration, data import
CronJobScheduled tasksNightly reports, cleanup

Service & Ingress — Networking

Service

Stable endpoint cho nhóm Pods (vì Pod IPs thay đổi khi recreate). Service cung cấp DNS name + load balancing.

TypeMô tảUse case
ClusterIP (default)Internal IP, chỉ accessible trong clusterService-to-service communication
NodePortExpose trên mỗi Node IP:PortDevelopment, testing
LoadBalancerExternal load balancer (cloud provider)External-facing services
ExternalNameDNS CNAME recordExternal database aliasing

Ingress

HTTP/HTTPS routing rules — map domain names/paths đến Services. Cần Ingress Controller (nginx, Traefik, HAProxy).

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: crm-ingress
  annotations:
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
spec:
  tls:
  - hosts:
    - crm.bank.vn
    secretName: crm-tls
  rules:
  - host: crm.bank.vn
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: crm-api
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: crm-frontend
            port:
              number: 80

ConfigMap & Secret

ConfigMap

Lưu trữ non-sensitive configuration data dưới dạng key-value. Decouple config khỏi image.

  • Inject vào Pod qua environment variables hoặc volume mounts
  • Thay đổi config không cần rebuild image
  • Mounted ConfigMaps tự động update (sau vài giây)

Secret

Giống ConfigMap nhưng cho sensitive data (passwords, tokens, certificates). Base64 encoded (NOT encrypted by default).

Security Kubernetes Secrets mặc định chỉ base64 (NOT encrypted). Cần enable encryption at rest trong etcd. Production nên dùng external secret manager (HashiCorp Vault, AWS Secrets Manager) với External Secrets Operator.

Storage

Volume Model

StorageClass → PersistentVolume (PV) → PersistentVolumeClaim (PVC) → Pod
ResourceVai trò
StorageClassDefines HOW storage is provisioned (SSD, HDD, network, local)
PersistentVolume (PV)A piece of storage trong cluster (provisioned by admin hoặc dynamically)
PersistentVolumeClaim (PVC)Request cho storage bởi Pod — bind đến PV matching requirements

Access Modes

  • ReadWriteOnce (RWO) — 1 node mount read-write (databases)
  • ReadOnlyMany (ROX) — nhiều nodes mount read-only (shared config)
  • ReadWriteMany (RWX) — nhiều nodes mount read-write (shared files — NFS, CephFS)

Autoscaling & Rolling Updates

Horizontal Pod Autoscaler (HPA)

Tự động scale số replicas dựa trên metrics (CPU, memory, custom metrics).

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: crm-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: crm-api
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

Vertical Pod Autoscaler (VPA)

Tự động adjust resource requests/limits cho pods. Hữu ích khi không biết app cần bao nhiêu resources.

Rolling Update Strategy

Default strategy cho Deployment — update pods dần dần, zero downtime:

  • maxSurge — bao nhiêu pods mới có thể tạo thêm (default 25%)
  • maxUnavailable — bao nhiêu pods có thể unavailable (default 25%)
  • Kết hợp readinessProbe — pod mới phải ready trước khi tiếp tục
  • Rollback — kubectl rollout undo quay lại version trước
Case Study: CRM Service scaling CRM API dùng HPA: min 3 replicas (normal), max 20 (peak). Khi CPU > 70% → scale up. Dịp Tết traffic x5 → auto scale lên 15 pods trong 2 phút. Sau peak → scale down dần. Zero manual intervention.
