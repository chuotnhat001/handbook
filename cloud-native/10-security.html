Mục Tiêu Học Tập

Sau khi hoàn thành chương này, bạn sẽ: Hiểu mô hình RBAC trong Kubernetes. Nắm OAuth2/OIDC flow cho microservices authentication. Biết dùng Vault quản lý secrets. Hiểu image scanning, network policies, và zero-trust security model.

RBAC — Role-Based Access Control

Kubernetes RBAC

Kiểm soát WHO có thể làm WHAT trên WHICH resources trong cluster.

4 objects cốt lõi

ObjectScopeVai trò
RoleNamespaceĐịnh nghĩa permissions (verbs + resources) trong 1 namespace
ClusterRoleCluster-widePermissions toàn cluster hoặc cross-namespace
RoleBindingNamespaceGắn Role/ClusterRole cho user/group/serviceaccount trong namespace
ClusterRoleBindingCluster-wideGắn ClusterRole cho subjects toàn cluster
# Role: CRM team chỉ quản lý deployments trong namespace crm-prod
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: crm-prod
  name: crm-deployer
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "update", "patch"]
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]

Best Practices RBAC

  • Least privilege — chỉ cấp quyền tối thiểu cần thiết
  • Namespace isolation — mỗi team/app có namespace riêng với Role riêng
  • Service accounts — mỗi workload có SA riêng, không dùng default
  • Audit regularly — review permissions định kỳ, revoke unused
  • No cluster-admin cho developers — chỉ platform team

OAuth2 & OIDC

OAuth2 — Authorization

Protocol cho phép ứng dụng (client) truy cập resources thay mặt user mà không cần user share password.

OIDC — Authentication layer trên OAuth2

OpenID Connect thêm identity layer: ID Token (JWT) chứa user info (who you are), bổ sung cho Access Token (what you can do).

Flow cho Microservices

User login → Identity Provider (Keycloak) → JWT Token → API Gateway validates → Service receives claims

Token Types

TokenMục đíchLifetime
ID TokenIdentity — ai đang login (JWT với user claims)Short (5-15 min)
Access TokenAuthorization — quyền truy cập resourcesShort (15-60 min)
Refresh TokenLấy token mới không cần re-loginLong (hours-days)

Keycloak — phổ biến cho ngân hàng

  • Open-source Identity and Access Management
  • Supports OIDC, SAML, LDAP/AD integration
  • Tích hợp OKD/OpenShift native
  • Multi-tenancy, fine-grained permissions
  • Self-hosted — dữ liệu không ra ngoài (compliance)

HashiCorp Vault — Secrets Management

Tại sao cần Vault

  • Kubernetes Secrets chỉ base64 (không encrypted mặc định)
  • Secrets hardcode trong code/config = security disaster
  • Cần rotation, audit trail, dynamic secrets, encryption as service

Core Features

FeatureMô tả
Secret StorageEncrypted at rest, accessed via API/CLI
Dynamic SecretsGenerate DB credentials on-demand, auto-expire
Encryption as ServiceTransit engine — encrypt/decrypt data without storing
Leasing & RenewalSecrets có TTL, auto-revoke khi expire
AuditLog mọi access — ai đọc secret nào, khi nào

Vault + Kubernetes Integration

  • Vault Agent Sidecar — inject secrets vào pod filesystem
  • CSI Provider — mount secrets qua Container Storage Interface
  • External Secrets Operator — sync Vault secrets → K8s Secrets
Case Study ngân hàng DB passwords cho CRM, Payment, Loyalty services đều trong Vault. Dynamic secrets: mỗi pod startup → Vault generate unique DB credential (TTL 1 giờ) → auto-rotate. Nếu pod bị compromise, credential tự expire. Audit log cho compliance.

Admission Control

Block images với critical vulnerabilities khỏi chạy trong cluster:

  • OPA Gatekeeper — policy engine cho K8s admission
  • Kyverno — K8s-native policy engine (YAML-based, dễ hơn OPA)
  • Policy ví dụ: "Chỉ cho phép images từ harbor.bank.vn, phải signed, không có critical CVE"

Network Policy

Default: Allow All

Mặc định K8s cho phép mọi pod communicate với nhau. Network Policy cho phép restrict traffic.

Zero-Trust Networking

  • Default deny all — không traffic nào được phép trừ khi explicitly allow
  • Whitelist approach — chỉ mở đúng ports/protocols cần thiết
  • Namespace isolation — CRM pods không thể gọi trực tiếp Payment pods
# Default deny all ingress trong namespace crm-prod
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: crm-prod
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
---
# Cho phép crm-api nhận traffic từ ingress controller
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-to-api
  namespace: crm-prod
spec:
  podSelector:
    matchLabels:
      app: crm-api
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: ingress-system
    ports:
    - port: 8080

Zero Trust Architecture

"Never trust, always verify" — Mọi request phải authenticated và authorized, bất kể source (internal hay external).

Principles

  • Verify explicitly — authenticate mọi request (không trust internal network)
  • Least privilege access — minimum permissions, just-in-time access
  • Assume breach — segment network, limit blast radius, encrypt everything

Implementation trong K8s

LayerControl
NetworkNetwork Policies + Service Mesh mTLS (mutual TLS giữa mọi service)
IdentityService accounts + SPIFFE/SPIRE cho workload identity
AccessRBAC + admission controllers + policy engines
DataEncryption at rest (etcd) + in transit (TLS everywhere)
ObserveAudit logs + runtime detection (Falco) + compliance scanning

Supply Chain Security

Threats

  • Compromised base images (malware trong public Docker Hub images)
  • Dependency confusion attacks (malicious packages)
  • Tampered artifacts in CI/CD pipeline

Mitigations

  • Image signing — Cosign/Sigstore: verify image integrity trước deploy
  • SBOM (Software Bill of Materials) — track mọi dependencies
  • Trusted registries only — admission policy block untrusted sources
  • Reproducible builds — same source → same artifact
  • SLSA framework — Supply-chain Levels for Software Artifacts (maturity levels)
Compliance cho ngân hàng PCI DSS, SOC2, Thông tư 09 NHNN yêu cầu: encryption at rest + transit, access audit logs, vulnerability scanning, incident response plan. Vault audit + Network Policies + image scanning + RBAC = foundation cho compliance.
