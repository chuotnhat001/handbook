Oracle TimesTen In-Memory Database Cache

In-memory relational database — cache layer cho Oracle Database với microsecond latency

Nguồn: docs.oracle.com/timesten — Oracle Documentation

Oracle TimesTen là relational database được tối ưu hoàn toàn cho bộ nhớ RAM. Có thể hoạt động standalone (độc lập) hoặc làm Application-Tier Database Cache cho Oracle Database — đặt ở tầng ứng dụng để giảm latency xuống microsecond.

Tư duy cốt lõi: Thay vì để ứng dụng truy vấn trực tiếp Oracle Database qua mạng và disk I/O, TimesTen cache một tập con dữ liệu hot vào RAM ngay tại application tier. Ứng dụng đọc/ghi local memory — TimesTen tự động đồng bộ với Oracle Database phía sau.

Kiến trúc

In-Memory Storage Model

Toàn bộ dataset được nạp vào RAM khi database khởi động. Khác biệt cơ bản với disk-based database:

  • Không có disk I/O trong data path thông thường
  • Data structures tối ưu cho memory access patterns, không phải disk block I/O
  • Kernel DBMS overhead nhỏ hơn đáng kể so với traditional database

ACID & Persistence

Dù là in-memory, TimesTen vẫn đảm bảo đầy đủ ACID. Persistence qua transaction log files và checkpoint files trên disk — đây chỉ là durability path, không phải data path chính.

APIs hỗ trợ

SQL, ODBC, JDBC, OCI, ODP.NET, PL/SQL — tương thích cao với Oracle Database, giúp migration ứng dụng dễ dàng.

Cache Architecture

Application → TimesTen (RAM) → Cache Agent → Oracle Database

Ứng dụng kết nối trực tiếp đến TimesTen (shared memory hoặc client-server). TimesTen đồng bộ với Oracle Database qua cache agent process.

Cache Groups — Khái niệm trung tâm

Cache group là đơn vị cơ bản của TimesTen Cache. Mỗi cache group chứa một hoặc nhiều bảng TimesTen ánh xạ tới bảng tương ứng trong Oracle Database. Cache group xác định:

  • Dữ liệu nào được cache
  • Chiều propagation — chỉ đọc, ghi async, hay ghi sync
  • Cách refresh — tự động hay thủ công

Read-Only Cache Groups

Dữ liệu trong TimesTen là read-only — ứng dụng chỉ đọc. Thay đổi thực hiện tại Oracle Database, sau đó refresh vào TimesTen.

Cơ chế refresh: AUTOREFRESH với mode INCREMENTAL (chỉ refresh hàng thay đổi) hoặc FULL (toàn bộ bảng). Interval cấu hình theo giây/phút.

Use case: Reference data, lookup tables, danh mục sản phẩm, tỷ giá — dữ liệu thay đổi ít nhưng đọc nhiều.

AWT — Asynchronous WriteThrough

Ứng dụng ghi vào TimesTen → commit ngay lập tức → TimesTen propagate bất đồng bộ về Oracle qua cache agent.

Ưu điểmRủi ro
Throughput cao nhất trong các loại write-backWindow không nhất quán ngắn nếu TimesTen crash trước khi propagate
Hỗ trợ parallel propagationCần monitor transaction log file threshold
SQL array processing (batch operations)Propagation lag cần theo dõi

Use case: Telecom billing, order management, financial trading — cần tốc độ ghi cao.

SWT — Synchronous WriteThrough

Ghi phải được confirm tại cả TimesTen và Oracle trước khi commit trả về cho ứng dụng.

Lưu ý: Consistency mạnh nhất nhưng throughput thấp hơn AWT. SWT KHÔNG hỗ trợ Active Standby Pair replication.

Use case: Tài chính, ngân hàng — tính nhất quán tuyệt đối quan trọng hơn throughput.

User-Managed Cache Groups

Admin kiểm soát thủ công. Mỗi bảng trong group có thể là READONLY hoặc PROPAGATE:

  • LOAD CACHE GROUP — nạp dữ liệu từ Oracle (hỗ trợ WHERE clause)
  • REFRESH CACHE GROUP — cập nhật cache từ Oracle
  • FLUSH CACHE GROUP — ghi thay đổi về Oracle
  • UNLOAD CACHE GROUP — xóa dữ liệu khỏi cache

Dynamic Load

Cache dữ liệu theo demand (lazy loading). Không cần load toàn bộ bảng — hữu ích cho bảng rất lớn.

Passthrough levels (0–3): kiểm soát hành vi khi dữ liệu chưa có trong cache. Level 0 trả lỗi, level cao hơn tự động chuyển query sang Oracle.

Replication & High Availability

Active Standby Pair

Cơ chế HA chính: một database active, một standby. Active replicate transactions sang standby real-time. Khi active fail, standby promote lên active.

Cache Group TypeHỗ trợ Active Standby?
Read-Only✅ Có
AWT✅ Có
SWT❌ Không
User-Managed❌ Không

Return Services

  • RETURN RECEIPT — Active chờ standby nhận transaction
  • RETURN TWOSAFE — Active chờ standby confirm commit (mạnh nhất)
  • RETURN NOREPLY — Không chờ, throughput cao nhất, có thể mất data khi failover

Cài đặt & Cấu hình cơ bản

Yêu cầu

  1. Oracle Database user với đủ privileges
  2. TimesTen cache administration user
  3. Credentials: Oracle Wallet hoặc connection attributes
  4. tnsnames.ora cấu hình net service name trỏ đến Oracle Database
  5. TimesTen DSN (Data Source Name)

Tạo Cache Group

-- 1. Cấu hình cache agent credentials
CALL ttCacheUidPwdSet('oracle_user', 'password');

-- 2. Start cache agent
CALL ttCacheStart;

-- 3. Tạo read-only cache group
CREATE READONLY CACHE GROUP cg_products
AUTOREFRESH INTERVAL 30 SECONDS
FROM oracle_schema.products (
  product_id NUMBER NOT NULL PRIMARY KEY,
  product_name VARCHAR2(100),
  price NUMBER
);

-- 4. Load data từ Oracle
LOAD CACHE GROUP cg_products COMMIT EVERY 1000 ROWS;
-- Tạo AWT cache group
CREATE ASYNCHRONOUS WRITETHROUGH CACHE GROUP cg_orders
FROM oracle_schema.orders (
  order_id NUMBER NOT NULL PRIMARY KEY,
  customer_id NUMBER,
  amount NUMBER
);

-- Cấu hình Active Standby Pair
CREATE ACTIVE STANDBY PAIR active_dsn, standby_dsn
RETURN RECEIPT
CACHE GROUP cg_products
CACHE GROUP cg_orders;

Memory Management

  • Implement aging policies: LRU hoặc time-based aging để tự động free memory
  • Chỉ cache subset dữ liệu thực sự cần thiết cho hot path
  • Monitor tablespace usage thường xuyên

AWT Performance Tuning

  • Bật parallel propagation để tăng throughput ghi về Oracle
  • Tăng batch size cho SQL array processing
  • Monitor transaction log — nếu log tích tụ nhanh, tăng propagation workers
  • Defragment change log tables định kỳ

Autorefresh Optimization

  • Dùng INCREMENTAL thay vì FULL cho bảng lớn
  • Chọn interval phù hợp — không phải lúc nào cũng cần real-time
Quan trọng: Luôn deploy Active Standby Pair cho production. Chỉ dùng AWT hoặc Read-Only cache groups nếu cần replicate. Test failover scenario trước khi đưa vào production.

Monitoring

  • Cache agent status — set start policy always để tự restart
  • Autorefresh statistics — đảm bảo refresh không bị delay
  • Transaction log file levels — cho AWT, tránh đầy disk
  • Propagation lag — giữa TimesTen và Oracle
  • Memory usage & aging — theo dõi cache hit ratio
