GSD Framework

Git. Ship. Done — Framework lightweight quản lý context cho AI-driven development workflows

Nguồn: github.com/open-gsd/gsd-core — MIT License, 5.3k stars

GSD Core là framework mã nguồn mở được thiết kế để nâng cao AI-driven development workflows trên nhiều nền tảng khác nhau — Claude Code, Gemini CLI, Copilot, Cursor, Windsurf và nhiều công cụ khác. Framework phân phối công việc sang các subagent có fresh context để duy trì chất lượng đầu ra ổn định xuyên suốt cả session làm việc dài.

Tư duy cốt lõi: Thay vì nhét toàn bộ công việc vào một AI session và chịu đựng "context rot", GSD chia nhỏ công việc thành các milestone độc lập, mỗi executor nhận ~200k token fresh context để thực thi song song — chất lượng được duy trì ổn định từ đầu đến cuối project.

Vấn đề: Context Rot

Khi làm việc với AI coding assistant trong session dài, một vấn đề phổ biến xảy ra: chất lượng output giảm dần theo thời gian. Hiện tượng này được gọi là context rot — sự suy giảm chất lượng tích lũy khi AI lấp đầy context window của mình.

Biểu hiện của context rot:

  • AI bắt đầu "quên" các quyết định đã đưa ra trước đó trong session
  • Output không còn nhất quán với các phần đã viết trước
  • Chất lượng code giảm dần ở giai đoạn cuối của một session dài
  • AI phải "tóm tắt lại" context, mất đi nhiều chi tiết quan trọng

Giải pháp của GSD

GSD Core giải quyết context rot bằng hai cơ chế:

  1. Structured artifacts — Lưu trữ quyết định, trạng thái và context dưới dạng file (STATE.md, CONTEXT.md) để persist thông tin qua các session thay vì phụ thuộc vào AI memory
  2. Fresh-context subagents — Phân phối công việc thực thi sang các subagent riêng biệt, mỗi agent nhận ~200k token context sạch, không bị "nhiễm" bởi lịch sử session
Kết quả: Session chính (orchestrator) giữ context gọn nhẹ bằng cách delegate công việc nặng sang subagents. Subagents hoạt động với fresh context, cho ra output chất lượng cao nhất quán dù project có phức tạp đến đâu.

Five-Phase Loop

GSD Core tổ chức công việc theo vòng lặp 5 phase, lặp lại cho mỗi milestone:

1. Discuss→ 2. Plan→ 3. Execute→ 4. Verify→ 5. Ship

Mỗi milestone (tính năng lớn, module, sprint) đi qua đủ 5 phase này trước khi kết thúc. Sau đó vòng lặp bắt đầu lại cho milestone tiếp theo.

Phase 1: Discuss

Ghi lại các quyết định implementation trước khi bắt đầu lập kế hoạch. Mục tiêu là capture tại sao một approach được chọn, không chỉ là cái gì sẽ build.

  • Làm rõ yêu cầu và constraints
  • Xác định các trade-off và quyết định architecture
  • Document vào CONTEXT.md để tham chiếu ở các phase sau
Tại sao cần Discuss trước Plan: Nếu bỏ qua phase này, các quyết định quan trọng chỉ tồn tại trong context window. Khi bắt đầu session mới hoặc spawn subagent mới, những quyết định đó sẽ mất — gây ra inconsistency trong codebase.

Phase 2: Plan

Research và decompose công việc sao cho phù hợp với fresh context window của executor agents.

  • Nghiên cứu kỹ thuật, thư viện, API cần dùng
  • Phân rã thành các task cụ thể, có thể thực thi độc lập
  • Ước lượng scope của từng task để phù hợp với context window
  • Ghi kết quả vào PLAN.md
Nguyên tắc Plan: Mỗi task trong plan phải đủ nhỏ để một executor agent có thể hoàn thành trong một lần với ~200k token context — không cần gọi lại hay tham chiếu lịch sử dài.

Phase 3: Execute

Chạy các task song song. Mỗi executor agent nhận fresh context (~200k token sạch) và chỉ cần biết đúng phần task được giao.

  • Dispatch nhiều subagents chạy song song cho các task độc lập
  • Mỗi executor nhận context tối thiểu cần thiết: task description + relevant code + plan
  • Không accumulate lịch sử session trong executor
Sequential (truyền thống)Parallel với GSD
Một AI session xử lý hết — context tích lũyNhiều subagent chạy song song — mỗi agent fresh
Chất lượng giảm dần cuối sessionChất lượng đồng đều ở tất cả tasks
Không thể scaleScale tuyến tính theo số lượng task

Phase 4: Verify

Validate code đã build và diagnose/fix issues trước khi hoàn thành phase. Đây là gate quan trọng — không được skip.

  • Review output của các executor agents
  • Chạy tests, kiểm tra integration
  • Diagnose lỗi và fix trong phase này (không carry sang phase sau)
  • Cập nhật STATE.md với trạng thái hiện tại
Anti-pattern: Chuyển sang phase Ship khi Verify chưa pass. GSD yêu cầu code phải thực sự hoạt động trước khi tạo PR — không ship code "probably works".

Phase 5: Ship

Tạo PR và chuyển sang milestone tiếp theo.

  • Tạo Pull Request với mô tả đầy đủ
  • Cập nhật STATE.md — đánh dấu milestone hoàn thành
  • Reset context cho milestone tiếp theo
  • Vòng lặp bắt đầu lại từ Discuss cho milestone mới

State Artifacts

GSD duy trì trạng thái project qua các file artifacts, giải quyết vấn đề mất context giữa các session:

ArtifactMục đíchAi cập nhật
STATE.mdTrạng thái hiện tại của project — phase nào, milestone nào, blockersCập nhật sau mỗi phase
CONTEXT.mdQuyết định, constraints, assumptions đã được documentCập nhật trong Discuss phase
PLAN.mdTask breakdown cho milestone hiện tạiTạo trong Plan phase

STATE.md

File trung tâm để resume công việc sau khi context bị mất (restart session, switch agent). Chứa:

  • Milestone hiện tại đang làm
  • Phase hiện tại trong five-phase loop
  • Tasks đã hoàn thành / đang làm / chưa làm
  • Blockers hoặc vấn đề cần giải quyết
  • Quyết định quan trọng đã được đưa ra
Khi bắt đầu một session mới, AI đọc STATE.md trước — đây là "bản đồ" để biết mình đang ở đâu trong project mà không cần phải recap toàn bộ lịch sử.

CONTEXT.md

Lưu trữ các quyết định và context quan trọng được document trong Discuss phase:

  • Lý do chọn architecture hiện tại
  • Constraints và assumptions của project
  • Các approach đã bị loại bỏ và lý do
  • Dependencies và integration requirements

Installation

Cài đặt nhanh (npm)

npx @opengsd/gsd-core@latest

Installer sẽ hỏi về runtime và installation preference (global hoặc local). Với môi trường không dùng Node.js, GSD cung cấp hướng dẫn cài đặt riêng cho từng runtime.

Khởi tạo project mới

/gsd-new-project

Command này khởi tạo cấu trúc artifacts và hướng dẫn bắt đầu five-phase loop cho project mới.

Lưu ý môi trường: GSD được thiết kế để hoạt động với nhiều AI runtime khác nhau. Installer tự detect runtime và cung cấp hướng dẫn phù hợp. Với non-Node.js environments, xem runtime-specific installation guides trên repo.

Onboarding existing codebase

GSD cũng hỗ trợ onboarding các codebase hiện có — không chỉ dành cho project mới. Quy trình tương tự: đọc codebase hiện tại trong Discuss phase để document existing architecture và decisions vào CONTEXT.md, sau đó proceed bình thường với five-phase loop.
