Anthropic Skills

Agent Skills — hệ thống folder-based instructions mở rộng năng lực Claude cho các tác vụ chuyên biệt, lặp lại và nhất quán

Nguồn: github.com/anthropics/skills | agentskills.io/specification — Apache 2.0 License

Agent Skills là một feature của Claude Code và Claude.ai cho phép tải các “skills” — các folder chứa instructions, scripts và resources — để Claude thực hiện các tác vụ chuyên biệt một cách lặp lại, nhất quán. Repo anthropics/skills là bộ sưu tập skills chính thức của Anthropic, phát hành dưới Apache 2.0 license (document skills là source-available).

Tư duy cốt lõi: Skill là cách “dạy” Claude một quy trình cụ thể. Thay vì viết prompt dài mỗi lần, bạn đóng gói instructions thành một folder tự chứa — Claude đọc và thực hiện theo workflow đã định sẵn, cho kết quả nhất quán qua nhiều session. Mỗi skill là một “chuyên gia” trong lĩnh vực hẹp của nó.

Cách hoạt động

Khi một skill được kích hoạt, Claude đọc file SKILL.md và tuân theo các instructions trong đó để hoàn thành tác vụ. Hệ thống hoạt động theo cơ chế 3 lớp:

  • Instructions — hướng dẫn Claude cách thực hiện tác vụ (SKILL.md body)
  • Scripts — các helper scripts (Python, Shell, JS) hỗ trợ workflow tự động
  • Resources — reference materials, examples, templates, assets
User invoke skill→ Claude đọc SKILL.md→ Tải scripts/resources nếu cần→ Thực thi theo instructions→ Output nhất quán

Claude quyết định khi nào invoke một skill dựa trên field description trong frontmatter. Khi task của user khớp với mô tả của skill, Claude tự động kích hoạt nó — hoặc user có thể gọi trực tiếp bằng slash command.

Progressive Disclosure (3 cấp tải)

Skills được thiết kế theo nguyên tắc progressive disclosure để tối ưu token usage. Không phải toàn bộ nội dung skill được tải cùng lúc:

CấpTokenNội dungKhi nào tải
Metadata~100 tokensname + descriptionLúc khởi động — tải cho TẤT CẢ skills đã cài
Instructions< 5000 tokens (khuyến nghị)Full SKILL.md bodyKhi skill được activate/invoke
ResourcesOn-demandFiles trong scripts/, references/, assets/Chỉ tải khi Claude cần trong quá trình thực thi
Tại sao progressive disclosure quan trọng: Với ~100 tokens/skill cho metadata, bạn có thể cài hàng chục skills mà không ảnh hưởng đến performance. Chỉ khi skill được kích hoạt, nội dung chi tiết mới vào context window. Điều này cho phép “skill library” lớn mà vẫn giữ context lean.

Cấu trúc thư mục Skill

Tối thiểu (chỉ cần 1 file)

my-skill/
└── SKILL.md          # File bắt buộc — instructions chính

Đầy đủ (với scripts và resources)

skill-name/
├── SKILL.md          # Bắt buộc: metadata + instructions
├── scripts/          # Tùy chọn: executable code (Python, Shell, JS)
├── references/       # Tùy chọn: tài liệu tham khảo
├── assets/           # Tùy chọn: templates, images, resources
└── LICENSE.txt
Quy tắc quan trọng: Tên thư mục PHẢI KHỚP với giá trị name trong frontmatter của SKILL.md. Ví dụ: thư mục webapp-testing/ thì frontmatter phải có name: webapp-testing.

Viết SKILL.md hiệu quả

Một SKILL.md hoàn chỉnh bao gồm frontmatter và body instructions:

---
name: my-testing-skill
description: |
  Run comprehensive test suites for Python projects using pytest.
  Use when the user asks to test, validate, or check code quality.
license: MIT
compatibility: Requires pytest and coverage installed
---

# Python Testing Skill

## Overview
This skill runs pytest-based test suites with coverage reporting.

## Workflow
1. Detect test framework (pytest.ini, setup.cfg, pyproject.toml)
2. Run tests with coverage: `pytest --cov=src --cov-report=term-missing`
3. Report results with actionable feedback

## When tests fail
- Show the failing test name and assertion
- Show the relevant source code
- Suggest a fix based on the error pattern

## Coverage targets
- Minimum: 80% line coverage
- Report uncovered branches explicitly
Progressive disclosure trong SKILL.md: Bắt đầu với overview ngắn gọn, sau đó đi vào chi tiết. Claude đọc từ trên xuống — thông tin quan trọng nhất đặt ở đầu file. Giữ dưới 5000 tokens (khuyến nghị) để không chiếm quá nhiều context window.

Cấu trúc nội dung SKILL.md khuyến nghị

  1. Overview — 1-2 câu mô tả skill làm gì
  2. Workflow — các bước thực hiện theo thứ tự
  3. Decision points — khi nào làm gì (if/else logic)
  4. Output format — kết quả mong đợi trông như thế nào
  5. Edge cases — xử lý lỗi, fallback behavior
  6. References — trỏ đến files trong references/ hoặc scripts/ nếu cần

Skill Creator — Workflow 5 bước

skill-creator là meta-skill (skill để tạo skill). Nó hướng dẫn Claude qua 5 bước lặp (iterative loop) để tạo và tinh chỉnh một skill mới:

1. Capture Intent→ 2. Draft→ 3. Test & Evaluate→ 4. Iterate→ 5. Optimize Description

Bước 1: Capture Intent

Phỏng vấn user để hiểu rõ yêu cầu:

  • Skill sẽ làm gì cụ thể?
  • Khi nào nó nên được trigger (use cases)?
  • Expected outputs là gì (format, nội dung)?
  • Có test cases nào để đánh giá chất lượng?
  • Môi trường nào (Claude Code vs Claude.ai)?

Bước 2: Draft

  • Viết SKILL.md với frontmatter (name, description) và instructions
  • Giữ dưới ~500 lines để đảm bảo hiệu quả
  • Dùng progressive disclosure: thông tin quan trọng lên đầu
  • Giải thích “why” thay vì chỉ “what” — Claude xử lý edge cases tốt hơn khi hiểu động lực

Bước 3: Test & Evaluate

  • Spawn parallel test runs: with-skill vs baseline (không có skill)
  • Dùng eval-viewer (browser-based) để xem kết quả trực quan
  • Đánh giá theo các tiêu chí: độ chính xác, nhất quán, đầy đủ
  • Dùng quantitative assertions khi có thể

Bước 4: Iterate

  • Review kết quả với user, identify patterns trong feedback
  • Cải thiện skill: generalize từ examples, không overfit
  • Tập trung vào “why” — giải thích reasoning thay vì liệt kê rules cứng
  • Rerun tests vào iteration directories mới để so sánh

Bước 5: Optimize Description

  • Refine field description để Claude trigger đúng lúc
  • Description nên “a little bit pushy” — chủ động hơn để tránh undertriggering
  • Dùng pattern “Use when...” để chỉ rõ use cases
  • Kiểm tra undertriggering sớm — đây là lỗi phổ biến nhất

3 Agents hỗ trợ Evaluation

Skill Creator sử dụng 3 sub-agents chuyên biệt để đánh giá chất lượng skill trong quá trình test:

AgentVai tròOutput
grader.mdĐọc execution transcript, đánh giá output theo expectations đã định nghĩaJSON với scores và nhận xét
comparator.mdSo sánh mù (blind comparison) giữa output A vs B, chấm điểm content + structureStructured comparison report
analyzer.mdPhân tích kết quả comparison, tìm patterns và insightsSummary với recommendations

Quy trình evaluation:

Chạy test cases→ Grader chấm điểm→ Comparator so sánh→ Analyzer tổng hợp→ Iterate

Best Practices

  1. Giải thích “why” thay vì chỉ “what” — Explain lý do đằng sau instructions thay vì chỉ đưa ra rules cứng nhắc. Claude sẽ handle edge cases tốt hơn khi hiểu động lực.
  2. Giữ lean — Remove các elements không productive. Mỗi dòng trong SKILL.md nên có mục đích rõ ràng. Dưới 500 lines.
  3. Description quyết định invocation — Field description ảnh hưởng trực tiếp đến khi nào Claude invoke skill. Viết cụ thể, “a little bit pushy”.
  4. Kiểm tra undertriggering sớm — Lỗi phổ biến nhất là skill không được trigger khi cần. Test với nhiều cách diễn đạt khác nhau.
  5. Generalize, không overfit — Viết để handle diverse future use cases, không chỉ test cases hiện tại.
  6. Adapt theo environment — Claude.ai (không có subagents, không có browser) vs Claude Code (có terminal, có file system) cần instructions khác nhau.
  7. Bundle repeated work — Nếu nhiều test cases đều cần cùng một bước, đưa vào skill để tránh lặp lại.
Anti-pattern: Không viết SKILL.md như documentation cho người đọc. Skill là instructions cho Claude — viết như đang chỉ dẫn một đồng nghiệp thông minh: “Khi nhận task X, làm A, rồi B, rồi C. Nếu gặp vấn đề Y, xử lý bằng Z.”

Webapp Testing Skill

Skill webapp-testing cho phép Claude tự động test các web applications local bằng Playwright (Python). Đây là một trong những skill phức tạp nhất, minh họa rõ cách một skill hoạt động với scripts và decision logic.

Core Pattern: Reconnaissance-then-Action

1. Navigate→ 2. Wait→ 3. Inspect→ 4. Act
  1. Navigate — goto URL (file:// hoặc http://localhost)
  2. Wait — page.wait_for_load_state('networkidle') (bắt buộc với dynamic apps)
  3. Inspect — screenshot HOẶC dump DOM để hiểu trang
  4. Act — click, fill, submit dựa trên selectors đã xác định

Decision Tree

Loại appCách tiếp cận
Static HTMLDùng file:// URL trực tiếp, không cần server
Dynamic app (React, Vue...)Dùng with_server.py với --server và --port flags
Multi-service (backend + frontend)Nhiều --server flags cho từng service

Scripts đi kèm

# Server lifecycle management
python scripts/with_server.py --server "npm run dev" --port 5173 \
  -- python your_automation.py

# Multiple servers (backend + frontend)
python scripts/with_server.py \
  --server "python backend.py" --port 8000 \
  --server "npm run dev" --port 5173 \
  -- python your_automation.py

Example scripts trong skill

  • element_discovery.py — khám phá buttons, links, inputs trên trang
  • static_html_automation.py — tương tác với file HTML local
  • console_logging.py — bắt và ghi lại console logs trong browser

Key Practices

  • Dùng sync_playwright() (synchronous API, không phải async)
  • Luôn đóng browser sau khi xong (browser.close())
  • Dùng descriptive selectors: text=Button Label hoặc #id
  • Viewport mặc định: 1920x1080
  • Luôn wait for network idle trước khi inspect dynamic content

Frontend Design Skill

Skill frontend-design hướng dẫn Claude thiết kế frontend với triết lý cốt lõi: “Each project should feel like it was made for this subject, not like a template applied to any subject.”

2-Pass Workflow

PassMục đíchOutput
Pass 1: BrainstormLên kế hoạch compact: palette, typography, layout, signature elementDesign plan ngắn gọn
Pass 2: ReviewTự hỏi: “Plan này specific với project này không, hay generic?”Refined plan hoặc redo

Design Principles

  • Hero element: Mở đầu bằng “the most characteristic thing” của chủ đề — không phải generic hero section
  • Typography as personality: Font mang tính cách, không trung lập — font choice phải có lý do
  • Structural choices mang ý nghĩa: Mỗi decision trả lời “tại sao cái này ở đây?”
  • Motion sparingly: Chỉ để clarify flow — quá nhiều motion trông “AI-generated”
  • Restraint: Chọn 1 chỗ bold, mọi thứ còn lại quiet
  • Complexity matches vision: Không thêm complexity vì “trông xịn hơn”

Anti-patterns (tuyệt đối tránh)

Không được ship:
  • Default card grids không có hierarchy
  • Stock hero: centered headline + gradient blob + generic CTA
  • Unmodified library defaults (Tailwind/shadcn mặc định)
  • Flat layouts không có layering, depth, hoặc motion
  • Safe gray-on-white với 1 accent color duy nhất
  • Dashboard generic không có point of view
  • Default font stack không có lý do rõ ràng

Yêu cầu tối thiểu (4/10)

Mỗi frontend surface phải thể hiện ít nhất 4 trong 10 đặc điểm sau:

  1. Hierarchy rõ ràng qua scale contrast
  2. Rhythm có chủ đích trong spacing
  3. Depth/layering qua overlap, shadows, surfaces, hoặc motion
  4. Typography có character và pairing strategy
  5. Color được dùng semantically, không chỉ decorative
  6. Hover, focus, active states được thiết kế cẩn thận
  7. Grid-breaking editorial hoặc bento composition
  8. Texture, grain, hoặc atmosphere khi phù hợp
  9. Motion clarify flow thay vì distract
  10. Data visualization là phần của design system

MCP Builder Skill

Skill mcp-builder hướng dẫn Claude xây dựng Model Context Protocol (MCP) servers. Triết lý: “The quality of an MCP server is measured by how well it enables LLMs to accomplish real-world tasks.”

4 Phases

PhaseMục đíchOutput
1. Research & PlanningNghiên cứu MCP spec, chọn tech stackArchitecture plan
2. ImplementationProject structure, API clients, error handling, paginationWorking MCP server code
3. Review & TestingCode quality, build verification, MCP InspectorTested, reviewed code
4. Evaluation10 câu hỏi test thực tế (độc lập, read-only, phức tạp, verifiable)Evaluation report

Tech Stack khuyến nghị

Ngôn ngữSDK/FrameworkTransportNaming Convention
TypeScript (khuyến nghị)@modelcontextprotocol/sdkstdio (local) hoặc StreamableHTTP (remote)Project: {service}-mcp-server, Tools: {service}_{action} (snake_case)
PythonFastMCPstdioTương tự, luôn dùng async/await

Patterns quan trọng

  • Pagination bắt buộc: Mọi tool trả về list phải có limit/offset
  • Character limits: Tránh context overflow bằng cách giới hạn response size
  • Error messages actionable: Hướng dẫn agent bước tiếp theo khi gặp lỗi
  • Tool descriptions đủ chi tiết: LLM phải tự quyết định được khi nào dùng tool nào
  • Zod schemas với .strict(): Validate input chặt chẽ (TypeScript)
  • Pydantic v2: Validate input (Python)
  • registerTool() pattern: Đăng ký tools theo pattern mới nhất
