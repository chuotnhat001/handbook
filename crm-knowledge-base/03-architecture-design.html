Phase 3: Architecture & Design

Enterprise, Solution, Data và Security Architecture cho CRM hệ thống cấp doanh nghiệp

4 domains · 16 modules · ~120 giờ học · Vai trò: Enterprise Architect, Solution Architect, Software Architect
Tư duy cốt lõi Architecture quyết định mọi constraint kỹ thuật. Thiết kế sai ở tầng này sẽ gây ra technical debt khổng lồ. Phase 3 áp dụng TOGAF, C4 Model, DDD và Clean Architecture để tạo blueprint vững chắc trước khi viết một dòng code nào.

Domain 3.1 — Enterprise Architecture

Kiến trúc tổng thể doanh nghiệp — đảm bảo CRM phù hợp với IT landscape hiện có.

Module 3.1.1 — TOGAF & Enterprise Framework

Mục đích module CRM không tồn tại trong chân không — nó phải fit vào IT landscape hiện có (ERP, eCommerce, billing, HR systems). Module này áp dụng TOGAF ADM để đảm bảo CRM decisions aligned với enterprise strategy, tránh "silo architecture" nơi mỗi team build theo cách riêng rồi 3 năm sau phải rewrite vì không integrate được.

DOC-301: Enterprise Architecture for CRM

Mục tiêu: Áp dụng EA framework cho CRM program Độ khó: Advanced Critical Thời gian: 12 giờ Tiêu chuẩn: TOGAF 10, ArchiMate 3.2

Các bước thực hiện:

  1. Xác định Architecture Vision — Trả lời: CRM sẽ fit vào enterprise landscape như thế nào trong 3-5 năm? Key decisions: build/buy/compose ratio, cloud strategy (full cloud vs hybrid), integration philosophy (API-first vs ETL). Publish Architecture Principles (8-12 guiding statements). Mục đích: north star cho mọi architecture decision tiếp theo — khi team debate microservices vs monolith, refer back to principles.
  2. Map Business Architecture — Sử dụng ArchiMate Business Layer: Value Streams (end-to-end flows delivering customer value), Business Capabilities (what the org CAN do), Business Processes (how it does them), Business Actors (who). Link CRM capabilities vào value streams. Mục đích: prove CRM supports business strategy, không phải "technology for technology's sake".
  3. Map Application Architecture — Document current Application Landscape: all systems touching customer data (CRM, ERP, website, mobile app, email platform, billing, analytics). Map data flows between them. Identify: redundancies, gaps, integration pain points. Mục đích: baseline để plan target state — know where you are before deciding where to go.
  4. Design Technology Architecture — Decisions: cloud provider(s), container orchestration, database technologies, messaging infrastructure, CDN, monitoring stack. Document as Technology Reference Model. Mục đích: standardize technology choices across CRM program — prevents 5 teams choosing 5 different message brokers.
  5. Establish Architecture Governance — Set up: Architecture Review Board (monthly review of key decisions), ADR (Architecture Decision Record) practice (document every significant decision with context, options considered, decision rationale), Exception Process (how to deviate from principles). Mục đích: architecture doesn't drift over time — governance prevents "boiling frog" degradation as teams take shortcuts.

Sơ đồ: ArchiMate Layered View (Business → Application → Technology), Application Landscape Map (current state), Capability Heat Map (maturity scoring), TOGAF ADM Cycle (tailored)

Deliverables: Architecture Vision Document (signed by CIO/CTO), Architecture Principles (published), Application Landscape Map, ADR Log (ongoing)

Công cụ: Archi (free ArchiMate tool), Sparx Enterprise Architect, LeanIX (SaaS), Mega HOPEX

Module 3.1.2 — Business Capability Modeling

Mục đích module Capability Map trả lời "WHAT does the business need to do?" (không phải HOW). Đây là bridge giữa business strategy và technology decisions — mỗi capability map lên 1+ services trong architecture. Module này giúp ra quyết định: capability nào build in-house (differentiating), capability nào mua (commodity), capability nào partner.

DOC-302: CRM Capability Map

Mục tiêu: Xây dựng capability model cho CRM Độ khó: Advanced High Thời gian: 8 giờ Phụ thuộc: DOC-301

Các bước thực hiện:

  1. Xây dựng Capability Hierarchy (L0-L3) — Decompose: L0 (CRM Platform), L1 (Sales Management, Marketing Management, Service Management, Data Management, Platform Services), L2 (sub-capabilities per L1, e.g., Sales → Lead Mgmt, Pipeline Mgmt, Forecasting), L3 (granular capabilities, e.g., Lead Mgmt → Capture, Score, Route, Nurture). Mục đích: exhaustive inventory of WHAT the system must do — drives service decomposition in Phase 3.2.
  2. Map Capabilities to Services/Systems — Cho mỗi L2/L3 capability: which system(s) currently provide it? Which will provide it in target state? Identify: coverage gaps (no system does this), overlaps (3 systems do the same thing), strategic capabilities (build custom) vs commodity (buy SaaS). Mục đích: rationalization — eliminate duplicate systems, identify build-vs-buy per capability.
  3. Create Heat Map Analysis — Score each capability on 3 dimensions: Maturity (1-5, how well does current implementation serve the need?), Strategic Importance (1-5, how much does this differentiate us?), Investment Need (1-5, how much do we need to invest?). Visualize as colored heat map. Mục đích: executive-friendly visualization that drives budget allocation — "where should we invest most?"
  4. Perform Technology Portfolio Rationalization — Based on capability mapping: identify systems to retire, systems to consolidate, gaps requiring new investment. Create transition plan with dependencies and timeline. Mục đích: reduce total cost of ownership by eliminating redundant tools, and focus investment on high-strategic-value capabilities.

Sơ đồ: Capability Map (3 levels, visual), Heat Map (color-coded maturity × importance), Technology Radar (adopt/trial/assess/hold)

Deliverables: Capability Model Document (complete hierarchy), Rationalization Report (retire/consolidate/invest decisions), Investment Roadmap (prioritized by heat map)

Công cụ: LeanIX (enterprise), Ardoq, Excel + conditional formatting (lightweight), Miro (collaborative workshops)

Domain 3.2 — Solution Architecture

Thiết kế giải pháp chi tiết — từ domain modeling đến API contracts.

Module 3.2.1 — Domain-Driven Design for CRM

Mục đích module DDD giải quyết complexity problem: CRM domain có nhiều concepts trùng tên nhưng khác nghĩa (VD: "Customer" trong Sales context khác với Service context). Module này partition system thành bounded contexts rõ ràng, mỗi context có language riêng và model riêng. Kết quả: team có thể work independently trên từng context mà không conflict.

DOC-303: DDD Strategic & Tactical Design

Mục tiêu: Áp dụng DDD để thiết kế CRM domain model Độ khó: Advanced Critical Thời gian: 12 giờ Tiêu chuẩn: DDD (Eric Evans), Implementing DDD (Vaughn Vernon)

Các bước thực hiện:

  1. Conduct EventStorming Workshop — Gather domain experts + developers. Sticky notes trên timeline: Domain Events (orange, past tense — "OpportunityCreated"), Commands (blue — "CreateOpportunity"), Aggregates (yellow — "Opportunity"), Read Models, Policies, External Systems. Duration: 2-3 sessions × 3 hours. Mục đích: shared understanding của toàn bộ business flow — cả business và tech cùng thấy big picture, discover hidden complexity.
  2. Identify Bounded Contexts — Từ EventStorming output, group related concepts: Sales Context (Lead, Opportunity, Quote, Pipeline), Marketing Context (Campaign, Audience, Channel, Content), Service Context (Case, Ticket, SLA, Knowledge), Billing Context (Invoice, Payment, Subscription), Identity Context (User, Account, Permission). Mục đích: mỗi context = potential service boundary = team boundary. Wrong boundary = coupling hell; right boundary = independent deployability.
  3. Create Context Map — Define relationships giữa contexts: Partnership (Sales ↔ Marketing, equal collaboration), Customer-Supplier (Billing downstream of Sales), Conformist (Identity upstream, others conform to its model), Anti-Corruption Layer (Legacy system → ACL → new system). Mục đích: explicit contracts giữa teams/services — biết ai phụ thuộc ai, change impact analysis trước khi modify API.
  4. Define Ubiquitous Language per Context — Cho mỗi bounded context, tạo glossary: term + definition + examples + anti-examples. VD: trong Sales "Account" = company entity; trong Billing "Account" = billing entity (may include multiple Sales Accounts). Mục đích: eliminate ambiguity — khi dev đọc code trong Sales Context, "Account" means exactly one thing.
  5. Design Tactical Patterns — Cho mỗi key aggregate: identify root entity, define invariants (rules that must always be true), list child entities and value objects, identify domain events emitted. VD: Opportunity aggregate has invariants (stage must progress forward, amount must match line items). Mục đích: enforce business rules IN the domain model (not in controllers or DB triggers) — impossible states become unrepresentable.
  6. Map Domain Events to Integration Events — Distinguish: internal domain events (within bounded context, in-process) vs integration events (published to other contexts via message bus). Define event schema, versioning strategy, backward compatibility rules. Mục đích: decouple contexts — Sales publishes "OpportunityClosed" event, Billing subscribes and creates Invoice independently.

Sơ đồ: Context Map (all contexts + relationships), Aggregate Diagrams (per context), Domain Event Flow (timeline), Bounded Context Canvas (per context)

Deliverables: Domain Model Document (complete), Context Map (visual + documented), Ubiquitous Language Glossary (per context)

Công cụ: EventStorming (Miro/physical wall), Context Mapper (DSL tool), PlantUML/Mermaid (diagrams)

Module 3.2.2 — API & Service Design

Mục đích module API là "contract" giữa CRM và mọi consumer (frontend, mobile, integrations, partners). Module này thiết kế API layer theo principles: consistent (naming, error handling), evolvable (versioning without breaking clients), secure (auth, rate limiting), và developer-friendly (documentation, SDKs). Bad API design = integration hell cho 5+ năm.

DOC-304: CRM API Architecture

Mục tiêu: Thiết kế API layer cho CRM platform Độ khó: Advanced Critical Thời gian: 10 giờ Tiêu chuẩn: OpenAPI 3.1, REST, GraphQL, gRPC

Các bước thực hiện:

  1. Thiết kế REST API Guidelines — Define standards: resource naming (plural nouns, nested for relationships: /accounts/{id}/contacts), HTTP methods mapping (GET=read, POST=create, PUT=replace, PATCH=update, DELETE=remove), response format (envelope with data + meta + errors), pagination (cursor-based for performance), filtering (field[operator]=value). Mục đích: consistency across 50+ endpoints — any developer familiar with 1 endpoint can predict behavior of all others.
  2. Design GraphQL Schema (nếu applicable) — Define types mirroring domain model, design queries (read) và mutations (write) separately. Implement: field-level authorization, query complexity analysis (prevent N+1 and DoS), DataLoader pattern for batching. Mục đích: frontend teams get exactly the data they need in one request — no over-fetching (REST problem) or under-fetching (multiple round-trips).
  3. Xây dựng API Versioning Strategy — Options: URL path versioning (/v1/contacts), header versioning (Accept: application/vnd.crm.v2+json), or query param (?version=2). Define: breaking change policy (what constitutes breaking?), deprecation timeline (announce → sunset → remove: 6 months minimum), migration guides per version. Mục đích: evolve API without breaking existing integrations — partners and customers build on your API, breaking them = trust destruction.
  4. Thiết kế API Gateway & Infrastructure — Gateway responsibilities: authentication (JWT validation), rate limiting (per tenant, per endpoint), request routing, response caching, request/response transformation, analytics/logging. Choose pattern: single gateway vs BFF (Backend-for-Frontend, separate gateway per client type). Mục đích: cross-cutting concerns centralized — each service doesn't re-implement auth or rate limiting.
  5. Xây dựng Developer Experience (DX) — Deliverables: interactive API documentation (try-it-out feature), SDKs in top languages (JavaScript, Python, C#), sandbox environment (safe to test, pre-populated data), webhook testing tools (simulate events), changelog + migration guides. Mục đích: time-to-first-successful-API-call < 15 minutes — good DX = adoption; bad DX = support tickets.

Sơ đồ: API Architecture Diagram (gateway → services), Sequence Diagrams (key flows with auth), API Gateway Topology (routing rules)

Deliverables: API Design Guidelines Document, OpenAPI 3.1 Specs (per service), API Governance Policy (review process for new endpoints)

Công cụ: Swagger/OpenAPI (spec), Postman (testing + docs), Stoplight (design-first), Kong/Apigee/AWS API Gateway (infrastructure)

Module 3.2.3 — Microservices & Modularity

Mục đích module "Microservices" không phải mặc định đúng — module này evaluate 3 options: Monolith (simple, fast to start), Modular Monolith (structured but single deployment), Microservices (independent deployment, high operational cost). Chọn dựa trên team size, expected scale, và operational maturity. Sai lầm phổ biến: startup 5 người chọn microservices → 80% effort vào infrastructure thay vì business logic.

DOC-305: CRM Service Architecture

Mục tiêu: Thiết kế service decomposition cho CRM Độ khó: Advanced High Thời gian: 10 giờ Phụ thuộc: DOC-303, DOC-304

Các bước thực hiện:

  1. Evaluate Architecture Style — Score 3 options against team/context: Monolith (team <10, MVP phase, simple deployment), Modular Monolith (team 10-30, clear boundaries needed, shared deployment ok), Microservices (team 30+, independent scaling per module, DevOps maturity high). Document decision as ADR with rationale. Mục đích: evidence-based choice — "monolith first, extract services when pain is proven" is usually correct for new CRM projects.
  2. Decompose Services by Bounded Context — Map bounded contexts (from DOC-303) to services. Each service: owns its data (no shared database), exposes API (from DOC-304), publishes events (domain events → integration events). Define service boundaries test: "can this service be developed, tested, and deployed independently?" Mục đích: right-sized services — too big = monolith in disguise; too small = distributed monolith (worst of both worlds).
  3. Design Inter-service Communication — Synchronous (REST/gRPC): for queries needing immediate response (get customer profile). Asynchronous (events via Kafka/RabbitMQ): for commands that can be eventually consistent (order placed → billing notified). Define: which interactions are sync vs async, timeout policies, retry strategies. Mục đích: prevent cascading failures — if Billing service is down, Sales service still works (async decoupling).
  4. Implement Saga Pattern — For distributed transactions spanning services (e.g., "create deal + reserve inventory + generate quote"): define compensation actions (rollback steps), choose orchestration (central coordinator) vs choreography (event-driven). Document happy path + all failure scenarios. Mục đích: maintain data consistency without distributed transactions (2PC doesn't scale) — every business operation has defined rollback path.
  5. Document with C4 Model — Create 4 levels: L1 System Context (CRM + external systems + users), L2 Container (applications, databases, message brokers within CRM), L3 Component (internal structure of each container/service), L4 Code (class diagrams for complex modules only). Mục đích: architecture documentation at appropriate detail level — executives see L1, developers see L3-L4, new team members walk L1→L2→L3 to onboard.
  6. Design Resilience Patterns — Implement: Circuit Breaker (stop calling failing service, fail fast), Retry with Exponential Backoff (transient errors), Bulkhead (isolate resources per service), Timeout (no indefinite waits), Fallback (degraded behavior when dependency unavailable). Mục đích: system stays operational during partial failures — one failing service doesn't take down entire CRM.

Sơ đồ: C4 Diagrams (all 4 levels), Service Interaction Diagram (sync + async flows), Saga Flow Diagram (happy path + compensation)

Deliverables: Architecture Decision Records (style choice + rationale), C4 Diagrams (maintained in code via Structurizr DSL), Service Catalog (all services with ownership, API, SLA)

Công cụ: Structurizr (C4 as code), IcePanel (interactive C4), Draw.io/Mermaid (quick diagrams), Backstage (service catalog)

Domain 3.3 — Data Architecture

Mô hình dữ liệu, integration patterns và event-driven architecture cho CRM.

Module 3.3.1 — Data Modeling & Storage

Mục đích module Data model là xương sống của CRM — mọi feature (search, reporting, personalization) phụ thuộc vào data được model đúng cách. Module này thiết kế 3 tầng (conceptual → logical → physical) và chọn storage technology phù hợp cho từng loại data. Sai lầm: dùng 1 RDBMS cho mọi thứ khi graph database phù hợp hơn cho relationship queries.

DOC-306: CRM Data Architecture

Mục tiêu: Thiết kế data model và storage strategy Độ khó: Advanced Critical Thời gian: 12 giờ Tiêu chuẩn: DAMA-DMBOK, ISO 8000

Các bước thực hiện:

  1. Xây dựng Conceptual Data Model — High-level entities và relationships (no attributes): Customer HAS many Contacts, Contact CREATES Opportunities, Opportunity CONTAINS Line Items, Case BELONGS TO Account. Validate với business stakeholders — đây là "business language" model. Mục đích: alignment giữa business và tech — business confirms "yes, this is how our world works" trước khi detail design.
  2. Thiết kế Logical Data Model — Add attributes, data types, constraints, cardinality. Design: primary keys (UUID vs sequential), foreign keys, indexes strategy, soft delete (is_deleted flag vs archive table). Handle: polymorphism (Activity table for calls, emails, meetings — STI vs separate tables), temporal data (effective dates, versioning). Mục đích: blueprint cho physical implementation — developers can generate schema from this, DBAs can optimize.
  3. Chọn Storage Technologies (Polyglot Persistence) — Match data type to storage: Relational (PostgreSQL — core entities, transactions, ACID), Document (MongoDB — flexible schemas, activity logs), Graph (Neo4j — relationship exploration "who knows who"), Search (Elasticsearch — full-text search, faceted filtering), Time-series (TimescaleDB — metrics, event streams), Cache (Redis — hot data, sessions). Mục đích: each technology optimized for its access pattern — don't force relational model onto graph problems.
  4. Design CQRS Architecture — Separate Write Model (normalized, enforces invariants, optimized for transactions) from Read Model (denormalized, optimized for queries, eventually consistent). Example: Write → "create opportunity with line items" (ACID transaction); Read → "dashboard showing pipeline by stage" (materialized view, updated via events). Mục đích: read and write have different optimization needs — CQRS lets each evolve independently.
  5. Thiết kế Event Sourcing (nếu applicable) — Store state changes as sequence of events (OpportunityCreated, StageChanged, AmountUpdated) instead of current state only. Rebuild current state by replaying events. Benefits: complete audit trail, temporal queries ("what was pipeline on March 1?"), event-driven integration. Mục đích: full history + perfect audit trail + ability to replay for analytics. Trade-off: more complex than CRUD — apply selectively to high-audit domains.
  6. Plan Data Partitioning & Sharding — Strategy for scale: horizontal partitioning by tenant_id (multi-tenant isolation), time-based partitioning for events/activities (archive old data), read replicas for reporting (offload analytics from OLTP). Define: partition key selection criteria, cross-partition query handling, rebalancing strategy. Mục đích: plan for 10x growth — don't redesign data layer when you hit 1M customers; design for it from day 1.

Sơ đồ: ERD (Conceptual, Logical, Physical — 3 separate diagrams), CQRS Architecture Diagram, Storage Decision Matrix (data type → technology mapping)

Deliverables: Data Model Document (all 3 levels), Schema Definitions (DDL scripts), Storage Technology ADR, Migration Strategy Document

Công cụ: ERwin/dbdiagram.io (modeling), DataGrip (schema management), Liquibase/Flyway (migrations), pgModeler (PostgreSQL-specific)

Module 3.3.2 — Integration & Event Architecture

Mục đích module CRM không hoạt động cô lập — nó phải exchange data với ERP (orders, invoices), eCommerce (purchases, carts), Marketing Tools (campaigns, engagement), HR (employee data), và nhiều hơn. Module này thiết kế integration backbone: event-driven, loosely-coupled, và resilient. Sai lầm phổ biến: point-to-point integrations tạo "spaghetti architecture" không maintain nổi khi có 20+ systems.

DOC-307: Integration Patterns & EDA

Mục tiêu: Thiết kế integration layer và event backbone Độ khó: Advanced High Thời gian: 10 giờ Tiêu chuẩn: Enterprise Integration Patterns (Hohpe), AsyncAPI

Các bước thực hiện:

  1. Inventory Integration Points — Map tất cả systems cần connect với CRM: direction (inbound/outbound/bidirectional), data exchanged, frequency (real-time/near-real-time/batch), volume (records per day), criticality (nếu integration fail, what business impact?). Mục đích: complete picture trước khi chọn technology — 3 real-time integrations vs 30 batch integrations = completely different architecture.
  2. Chọn Integration Patterns — Match pattern to use case: Request-Reply (synchronous, need immediate response — e.g., validate address), Publish-Subscribe (async, broadcast events to multiple consumers — e.g., "opportunity won" notifies billing + fulfillment), Event Sourcing (capture all state changes as events for downstream replay). Mục đích: avoid "one hammer for all nails" — each pattern has tradeoffs in latency, coupling, complexity.
  3. Select Message Broker Infrastructure — Evaluate: Apache Kafka (high throughput, replay, event streaming), RabbitMQ (routing flexibility, lightweight), NATS (ultra-low latency, edge). Decision criteria: throughput needs, ordering guarantees, retention requirements, operational complexity tolerance. Mục đích: infrastructure choice that scales with growth — don't choose RabbitMQ when you need Kafka's replay capability for event sourcing.
  4. Design Event Schema & Registry — Define event schema standard: CloudEvents spec (source, type, data, specversion). Implement Schema Registry for versioning (Avro/JSON Schema). Define compatibility rules: backward compatible (new version reads old data), forward compatible (old version reads new data). Mục đích: prevent "breaking changes in production" — producers can evolve schemas without breaking consumers.
  5. Implement Reliability Patterns — Design: Dead Letter Queue (DLQ — failed messages captured for investigation), Idempotency (consumers handle duplicate messages safely via idempotency key), Retry with backoff (transient failures auto-recover), Circuit Breaker (stop calling failing downstream). Mục đích: integration layer must be self-healing — temporary failures don't lose data or require manual intervention.
  6. Evaluate iPaaS vs Custom — When to use iPaaS (MuleSoft, Workato, Make): standard connectors exist, business users need to configure, quick time-to-value. When to build custom: high throughput requirements, complex transformation logic, tight coupling to domain events. Can coexist: iPaaS for 3rd-party SaaS integrations, custom for core domain events. Mục đích: cost-effective decision — don't build what MuleSoft does better, but don't pay MuleSoft pricing for simple webhook forwarding.

Sơ đồ: Integration Architecture (all systems + data flows), Event Flow Diagram (producers → broker → consumers), Message Schema Examples, Sequence Diagrams (per key integration)

Deliverables: Integration Architecture Document, Event Catalog (all events with schema + owner), AsyncAPI Specs (for event-driven interfaces)

Công cụ: Apache Kafka/Confluent (event streaming), MuleSoft/Workato (iPaaS), AWS EventBridge (serverless events), Debezium (CDC), Schema Registry

Domain 3.4 — Security Architecture

Bảo mật by design — IAM, encryption, compliance và threat modeling.

Module 3.4.1 — Identity & Access Management

Mục đích module CRM chứa dữ liệu nhạy cảm nhất của doanh nghiệp — thông tin khách hàng, deal amounts, contracts, payment info. Security breach = regulatory fine + reputation damage + customer trust destruction. Module này thiết kế defense-in-depth: authentication (ai đang access?), authorization (họ được phép làm gì?), encryption (data an toàn even if stolen), và audit (ai đã làm gì, khi nào?).

DOC-308: CRM Security Architecture

Mục tiêu: Thiết kế security cho CRM platform Độ khó: Advanced Critical Thời gian: 10 giờ Tiêu chuẩn: ISO 27001, OAuth 2.0, OWASP ASVS, Zero Trust

Các bước thực hiện:

  1. Thiết kế Authentication Architecture — Multi-protocol support: OAuth 2.0 + OIDC (modern web/mobile apps), SAML 2.0 (enterprise SSO integration), API Keys (service-to-service). MFA enforcement: TOTP, WebAuthn/FIDO2, SMS (fallback only). Session management: token rotation, idle timeout, absolute timeout. Mục đích: verify identity with certainty — stolen password alone should never be enough to access CRM data.
  2. Thiết kế Authorization Model — Layered approach: RBAC (Role-Based: Admin, Manager, Rep, Viewer), ABAC (Attribute-Based: own-records-only, territory-based), Field-Level Security (hide salary fields from non-HR), Record-Level Sharing (private → team → organization hierarchy). Implement: permission evaluation engine, caching strategy for permission checks. Mục đích: principle of least privilege — every user sees only what they need, nothing more.
  3. Design Data Protection & Encryption — At-rest: database encryption (TDE), field-level encryption (PII fields specifically), backup encryption. In-transit: TLS 1.3 everywhere, certificate management, internal mTLS between services. Key management: HSM-backed, key rotation schedule, separation of duties. Mục đích: even if attacker gains access to storage/network, data is unreadable without keys.
  4. Thiết kế Multi-tenant Isolation — Model options: shared database with row-level security (RLS), shared schema with tenant discriminator, separate schemas, separate databases. For each: isolation guarantees, performance impact, operational complexity. Choose based on: number of tenants, data sensitivity, compliance requirements. Mục đích: Tenant A can NEVER see Tenant B's data — this is existential for SaaS CRM.
  5. Conduct Threat Modeling (STRIDE) — For each component: identify threats using STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege). Draw data flow diagrams with trust boundaries. Rate: likelihood × impact. Define mitigations per threat. Mục đích: proactive security — find vulnerabilities in design before they're in code. Much cheaper to fix at architecture stage vs post-deployment.
  6. Design Audit & Security Monitoring — Capture: who did what, when, from where, on which record. Immutable audit log (append-only, tamper-evident). Real-time alerting: impossible travel, brute force attempts, privilege escalation, bulk data export. Retention: regulatory minimum (often 7 years). Mục đích: forensics capability (investigate breaches), compliance evidence (auditors need proof), and deterrence (users know actions are logged).

Sơ đồ: IAM Architecture (auth flows per client type), Role Hierarchy (RBAC model), Threat Model (STRIDE DFD with trust boundaries), Security Zones (network segmentation)

Deliverables: Security Architecture Document (comprehensive), Threat Model (per component), Access Control Matrix (roles × permissions × resources)

Công cụ: Keycloak (open-source IAM), Auth0/Okta (managed), Azure AD B2C (Microsoft ecosystem), HashiCorp Vault (secrets), OWASP ZAP/Burp Suite (security testing)

Module 3.4.2 — Compliance & Privacy by Design

Mục đích module Compliance không phải "checkbox exercise" sau khi build xong — nó phải baked into architecture từ đầu. Retrofitting GDPR compliance vào existing system costs 10x more than designing for it. Module này đảm bảo privacy, data residency, và regulatory requirements là first-class architectural concerns.

DOC-309: Compliance Architecture

Mục tiêu: Tích hợp compliance vào kiến trúc CRM Độ khó: Advanced High Thời gian: 8 giờ Tiêu chuẩn: GDPR, CCPA, SOC 2, ISO 27701

Các bước thực hiện:

  1. Áp dụng Privacy by Design Principles — Implement Cavoukian's 7 principles: proactive not reactive, privacy as default, privacy embedded into design, full functionality (no trade-offs), end-to-end security, visibility/transparency, respect for user privacy. Map each principle to specific architectural decisions. Mục đích: privacy is not a feature you add — it's a property of the architecture. Design decisions should make privacy violations structurally difficult.
  2. Conduct Data Protection Impact Assessment (DPIA) — For each data processing activity: what personal data? legal basis? purpose? retention period? sharing with third parties? technical safeguards? Identify high-risk processing (profiling, automated decisions, large-scale monitoring). Mục đích: regulatory requirement (GDPR Art. 35) AND practical risk identification — forces you to justify every piece of personal data you collect.
  3. Thiết kế Consent Management Architecture — Technical implementation: consent records (who consented, to what, when, how), purpose-based access control (data only accessible for consented purposes), consent withdrawal (immediate effect across all systems), consent audit trail (prove what was consented at any point in time). Mục đích: "can you prove this customer consented to marketing emails on March 3?" — if you can't answer with confidence, you have compliance risk.
  4. Implement Right to Erasure (Technical Design) — Architecture for "delete my data": identify all locations (primary DB, backups, caches, analytics, logs, third-party systems), soft-delete vs hard-delete per location, anonymization strategy (keep aggregated analytics but remove PII), verification process (confirm deletion complete). Mục đích: GDPR Article 17 requires deletion within 30 days — you need automated, tested process, not manual ad-hoc deletion.
  5. Design Data Residency & Sovereignty — For multi-region deployment: which data must stay in-region (EU customer data in EU), cross-border transfer mechanisms (Standard Contractual Clauses, adequacy decisions), encryption for data in transit between regions, tenant-level region assignment. Mục đích: legal requirement in many jurisdictions — wrong architecture = data flowing to wrong region = regulatory violation.
  6. Thiết kế Compliance Automation — Automate: data subject access requests (DSAR — export all data about a person), consent preference enforcement (suppress from marketing when opted out), retention policy execution (auto-archive/delete when period expires), compliance reporting (evidence generation for auditors). Mục đích: manual compliance doesn't scale — when you have 1M customers and 100 DSARs/month, automation is the only viable path.

Sơ đồ: Consent Flow Diagram (capture → store → enforce → withdraw), Data Lifecycle Diagram (create → use → archive → delete), Compliance Architecture (enforcement points)

Deliverables: DPIA Template (filled for core processing activities), Privacy Architecture Document, Compliance Checklist (per regulation)

Công cụ: OneTrust/TrustArc (consent management), Osano (cookie consent), custom DSAR automation, Open Policy Agent (policy enforcement)
