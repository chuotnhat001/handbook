Phase 4: Implementation

Development, Configuration, Testing và Deployment — từ code đến production

4 domains · 16 modules · ~140 giờ học · Vai trò: Software Architect, Developer, DevOps Engineer, QA Lead
Tư duy cốt lõi Implementation biến architecture thành sản phẩm thực. Phase này đòi hỏi kỷ luật kỹ thuật cao: Clean Code, TDD, CI/CD, và Infrastructure as Code. Mỗi quyết định implementation phải truy ngược được đến architecture decision (Phase 3).

Domain 4.1 — Development

Phát triển phần mềm CRM — backend services, frontend applications và mobile clients.

Module 4.1.1 — Backend Development

Mục đích module Backend là "engine room" — nơi business rules được enforce, data được persist, và integrations được orchestrate. Module này implement Clean Architecture cho phép thay đổi framework/database mà không ảnh hưởng business logic. Key principle: domain layer không phụ thuộc vào bất kỳ infrastructure nào.

DOC-401: CRM Backend Engineering

Mục tiêu: Xây dựng backend services cho CRM Độ khó: Advanced Critical Thời gian: 16 giờ Tiêu chuẩn: Clean Architecture, SOLID, 12-Factor App

Các bước thực hiện:

  1. Setup Project Structure (Clean Architecture) — Organize theo layers: Domain (entities, value objects, domain events, repository interfaces), Application (use cases/handlers, DTOs, validators), Infrastructure (DB implementations, external APIs, message bus), Presentation (controllers, middleware). Dependency rule: inner layers NEVER reference outer layers. Mục đích: testable business logic in isolation — swap PostgreSQL for MongoDB without touching a single domain class.
  2. Implement Domain Layer — Code DDD tactical patterns từ DOC-303: Aggregates (enforce invariants), Entities (identity-based), Value Objects (equality by value), Domain Events (record state changes), Specifications (encapsulate query logic). Write architecture tests verifying dependency rule. Mục đích: business rules expressed in code, not scattered across controllers/services — one place to find "how does opportunity stage transition work?"
  3. Implement Application Layer (CQRS) — Separate Commands (write — CreateOpportunity, ChangeStage) from Queries (read — GetPipelineDashboard, SearchContacts). Use MediatR/similar for handler dispatch. Add cross-cutting: validation (FluentValidation), authorization checks, audit logging via pipeline behaviors. Mục đích: thin, focused handlers — each does ONE thing. Easy to test, easy to trace, easy to add cross-cutting concerns without modifying handlers.
  4. Implement Infrastructure Layer — Repository implementations (EF Core/Dapper/TypeORM), event bus publishing, external service adapters (email, file storage, payment), caching (Redis), search indexing (Elasticsearch sync). Mục đích: all "messy" I/O concerns contained here — domain stays pure, infrastructure can be swapped/mocked for testing.
  5. Implement Multi-tenancy — Choose pattern: shared DB with tenant discriminator (automatic query filtering), separate schemas, or separate databases. Implement: tenant resolution (from JWT claim, subdomain, or header), automatic tenant scoping on all queries, tenant-aware caching, cross-tenant data leak prevention tests. Mục đích: foundational for SaaS — every query automatically scoped to current tenant, impossible to accidentally serve wrong tenant's data.
  6. Implement Audit Trail — Capture: who (user ID), what (entity + field changed), when (timestamp), where (IP, user agent), old value, new value. Implementation options: change tracking interceptor, event sourcing replay, dedicated audit table per entity. Immutable storage. Mục đích: compliance requirement (who changed deal amount from $50K to $500K?) + debugging (what sequence of changes led to this state?).

Sơ đồ: Clean Architecture Layers (dependency arrows), Class Diagrams (per bounded context), Sequence Diagrams (per critical use case)

Deliverables: Code Repository Structure (documented conventions), Coding Standards Document, Architecture Tests (dependency rule verification)

Công cụ: .NET 8/Java 21/Node.js/Go (language choice ADR), PostgreSQL (primary DB), Redis (cache), Docker (containerization), ArchUnit/NetArchTest (architecture tests)

Module 4.1.2 — Frontend Development

Mục đích module CRM UI là nơi users spend 8+ hours/day — performance, UX, và reliability trực tiếp impact productivity và adoption. Module này xây dựng scalable frontend: design system cho consistency, state management cho data freshness, real-time cho collaboration, và accessibility cho inclusivity. Poor frontend = poor adoption = CRM failure.

DOC-402: CRM Frontend Engineering

Mục tiêu: Xây dựng UI/UX cho CRM platform Độ khó: Advanced High Thời gian: 12 giờ Phụ thuộc: DOC-304 (API Design)

Các bước thực hiện:

  1. Xây dựng Design System & Component Library — Create: tokens (colors, spacing, typography), atoms (Button, Input, Badge), molecules (FormField, SearchBar, Card), organisms (DataTable, FilterPanel, Timeline). Document in Storybook with all variants and states. Mục đích: UI consistency across 50+ screens without designers reviewing every page — developers compose from proven components.
  2. Thiết kế State Management Architecture — Separate concerns: Server State (TanStack Query/SWR — cached API data, auto-refresh, optimistic updates), Client State (Zustand/signals — UI state, form drafts, preferences), URL State (search params — filters, pagination, active tab). Mục đích: no single "global store" that becomes unmaintainable — each state type has appropriate tool with built-in best practices.
  3. Implement Real-time Updates — WebSocket for live collaboration: see who's viewing same record, real-time field updates (Google Docs style for notes), presence indicators, notification toasts. Implement: connection management (auto-reconnect, backoff), conflict resolution (last-write-wins vs merge), selective subscriptions (only records currently viewed). Mục đích: CRM feels alive and collaborative — users see changes from colleagues instantly, no "refresh to see updates".
  4. Implement Accessibility (WCAG 2.1 AA) — Requirements: keyboard navigation (all actions reachable without mouse), screen reader support (ARIA labels, live regions), color contrast (4.5:1 text, 3:1 UI), focus management (modal traps, skip links), reduced motion support. Automated testing: axe-core in CI. Mục đích: legal requirement in many jurisdictions + inclusive design = larger user base + better UX for everyone (keyboard shortcuts benefit power users too).
  5. Optimize Performance — Implement: code splitting (route-based lazy loading), virtualization (render only visible rows in 10K-record lists), image optimization (responsive images, lazy loading), bundle analysis (webpack-bundle-analyzer, tree shaking). Set performance budget: LCP <2.5s, FID <100ms, bundle <300KB gzipped. Mục đích: CRM must feel fast for daily use — 100ms delay on page load × 50 loads/day = user frustration and workaround behaviors.

Sơ đồ: Component Architecture (atomic design hierarchy), State Flow (server/client/URL separation), Micro-frontend Topology (if applicable)

Deliverables: Design System Documentation (Storybook), Component Library (npm package), Performance Budget (measured in CI)

Công cụ: React/Next.js (or Angular/Vue — ADR choice), Storybook (component docs), Figma (design tokens), Playwright (E2E), axe-core (a11y)

Module 4.1.3 — Mobile CRM Development

Mục đích module Sales reps, field service technicians, và managers cần CRM on-the-go. Mobile CRM phải work offline (poor connectivity in field), sync seamlessly khi online, và leverage device capabilities (camera for business cards, GPS for check-ins, biometrics for quick unlock). Module này thiết kế mobile experience that's a productivity multiplier, not a crippled web view.

DOC-403: Mobile CRM Application

Mục tiêu: Phát triển mobile CRM cho field teams Độ khó: Advanced Medium Thời gian: 10 giờ Phụ thuộc: DOC-401, DOC-402

Các bước thực hiện:

  1. Chọn Platform Strategy — Evaluate: Native (Swift + Kotlin — best performance, full device access, 2x development cost), Cross-platform (Flutter/React Native — shared codebase 80%+, near-native performance, faster iteration), PWA (web-based, no app store, limited device access). Decision criteria: offline requirements, device API needs, team skills, maintenance budget. Mục đích: right tech for right constraints — don't over-engineer if PWA suffices, don't under-engineer if offline is critical.
  2. Thiết kế Offline-first Architecture — Local database (SQLite/Realm) mirroring server data for key entities. Sync strategy: pull on launch + push on change + periodic background sync. Conflict resolution: timestamp-based (last-write-wins for simple fields), merge (append for activity logs), manual (flag conflicts for user resolution on critical data). Mục đích: app fully functional without network — sales rep can view contacts, log calls, update opportunities even in elevator or airplane.
  3. Implement Push Notifications — Categories: actionable (deal assigned — tap to view), informational (weekly pipeline summary), urgent (SLA breach imminent — requires action). Architecture: server events → push service (FCM/APNs) → device → notification channels (grouped, priority). User preferences: per-category opt-in/out, quiet hours, snooze. Mục đích: keep users informed without overwhelming — right notification at right time drives engagement, too many = disabled = silent CRM.
  4. Integrate Device Capabilities — Camera: business card scanning (OCR → contact creation), document capture (attach to records). Location: check-in/check-out at customer sites, nearby accounts on map, mileage tracking. Calendar: sync CRM activities ↔ device calendar. Contacts: import phone contacts as leads. Mục đích: mobile-specific features that ADD value beyond "web in smaller screen" — reasons to prefer mobile over desktop for field activities.
  5. Plan Distribution & Security — Enterprise deployment: MDM (Mobile Device Management) integration, app configuration profiles, remote wipe capability. Security: biometric unlock (Face ID/fingerprint), certificate pinning, encrypted local storage, session timeout. App Store vs enterprise-only distribution. Mục đích: IT admin can manage fleet of devices, enforce security policies, and deploy updates without user action.

Sơ đồ: Mobile Architecture (offline + online layers), Offline Sync Flow (conflict resolution decision tree), Push Notification Architecture (server → cloud → device)

Deliverables: Mobile App Architecture Document, Sync Strategy Specification, Platform Guidelines (iOS + Android)

Công cụ: Flutter/React Native (cross-platform), Firebase (push + analytics), SQLite/Realm (local DB), Detox/Appium (mobile testing), Fastlane (CI/CD for mobile)

Domain 4.2 — Platform Configuration

Cấu hình nền tảng, workflow engine và extensibility framework.

Module 4.2.1 — Workflow & Business Rules Engine

Mục đích module Workflow engine là "brain" của CRM platform — automate mọi business process đã design ở Phase 2 mà không cần hardcode. Admin có thể thay đổi approval rules, SLA escalations, lead routing mà không deploy code mới. Module này build engine flexible đủ để handle hàng trăm workflows khác nhau, từ simple notifications đến complex multi-step approvals.

DOC-404: Workflow Engine Design

Mục tiêu: Xây dựng engine xử lý business workflows Độ khó: Advanced Critical Thời gian: 12 giờ Tiêu chuẩn: BPMN 2.0 execution, DMN

Các bước thực hiện:

  1. Chọn Engine Architecture — Evaluate: State Machine (simple, deterministic — good cho entity lifecycles like case status), BPMN Runtime (full process orchestration — good cho multi-step, multi-actor workflows), Temporal/Saga (distributed, long-running — good cho cross-service orchestration). Many CRMs need combination. Mục đích: right engine type cho right workflow complexity — don't force BPMN on simple field update triggers.
  2. Thiết kế Visual Workflow Builder — UI for admins to create workflows: drag-drop nodes (triggers, conditions, actions, delays, splits), connect with arrows, configure each node's properties. Preview mode (simulate execution). Version control (draft → published → archived). Mục đích: business users create workflows without developer involvement — key for agility and reducing IT backlog.
  3. Implement Trigger Types — Event-based (record created/updated/deleted, field changed to specific value), Time-based (scheduled — daily report, SLA approaching), Condition-based (field value matches rule — amount > $100K triggers approval), Manual (user clicks button). Support compound triggers (AND/OR combinations). Mục đích: workflows fire automatically when business conditions are met — no human remembering "I need to notify finance when deal > $100K".
  4. Build Business Rules Engine (DMN) — Implement Decision Tables: input columns (conditions to evaluate), output columns (actions/values to set). Support: hit policies (first match, collect all, priority), nested decisions (output of one table feeds into another). Visual table editor for business users. Mục đích: complex business logic (pricing rules, discount approvals, territory assignment) expressed as maintainable tables instead of if/else spaghetti code.
  5. Implement SLA Timers & Escalation — Timer types: response timer (start when case created, pause when first reply), resolution timer (start on assignment, pause on pending-customer), business hours calculation (exclude weekends, holidays per region). Escalation actions at thresholds (50% — warning, 80% — escalate to manager, 100% — breach notification to VP). Mục đích: SLA compliance automated — agents see countdown, managers get proactive alerts, customers experience consistent service levels.
  6. Design Error Handling & Compensation — Handle: action failures (email service down — retry with backoff), timeout (async step didn't complete — escalate or skip), compensation (if step 3 fails, undo step 1 and 2). Dead letter queue for permanently failed workflows. Admin dashboard for workflow health. Mục đích: workflows must be resilient — one failed notification shouldn't block an entire approval chain from proceeding.

Sơ đồ: Workflow Engine Architecture (components + interactions), State Machine Diagrams (entity lifecycles), DMN Decision Tables (example rules)

Deliverables: Workflow Engine Specification, Rule Templates (common patterns), Configuration Guide (for admins)

Công cụ: Camunda (BPMN), Temporal (distributed workflows), N8N (low-code automation), custom engine (for tight CRM integration)

Module 4.2.2 — Customization & Extension Framework

Mục đích module Mỗi doanh nghiệp có processes riêng — no CRM fits 100% out-of-box. Module này build platform extensibility: custom entities (thêm objects mới không cần code), custom fields (extend existing objects), plugins (add business logic), webhooks (integrate external systems). Goal: 80% customization by admin configuration, 20% by developer code.

DOC-405: Platform Extensibility

Mục tiêu: Thiết kế framework mở rộng cho CRM Độ khó: Advanced High Thời gian: 10 giờ Phụ thuộc: DOC-401, DOC-404

Các bước thực hiện:

  1. Thiết kế Custom Entity/Field Engine — Dynamic schema: admin tạo "custom object" (e.g., "Project", "Asset", "Warranty") với custom fields (text, number, date, lookup, formula, rollup). Storage: EAV pattern hoặc JSON columns hoặc dynamic DDL. Include: validation rules, default values, field dependencies. Mục đích: every business has unique data needs — real estate CRM needs "Property" entity, healthcare CRM needs "Patient" entity, without code changes.
  2. Build Plugin Architecture — Extension points: before/after CRUD operations (triggers), custom API endpoints (actions), scheduled jobs, UI widgets (custom components in record views). Isolation: each plugin runs in sandbox (WASM or container), resource limits (CPU, memory, execution time), no direct DB access (only via platform API). Mục đích: third-party developers extend CRM safely — one bad plugin cannot crash the platform or access other tenants' data.
  3. Implement Formula & Calculated Fields — Expression engine: support math (+, -, *, /), text (CONCAT, LEFT, SUBSTITUTE), logic (IF, AND, OR, CASE), date (DATEDIFF, TODAY, ADDDAYS), cross-object (related record fields). Real-time evaluation on read vs batch recalculation. Mục đích: business users create computed fields (e.g., "Days Since Last Contact = TODAY() - Last_Activity_Date") without developer tickets.
  4. Thiết kế Webhook & Event Subscriptions — Allow external systems to subscribe to CRM events: entity.created, entity.updated, entity.deleted, workflow.completed. Include: filtering (only fire for specific conditions), retry logic (exponential backoff on failure), payload configuration (which fields to include), signature verification (HMAC). Mục đích: push-based integration — external systems react to CRM events in real-time without polling.
  5. Build Sandbox & Testing for Extensions — Developer sandbox: isolated environment with sample data, no production impact. Testing tools: simulate events, mock external calls, assert outcomes. Deployment pipeline: sandbox → review → staging → production. Rollback capability. Mục đích: safe development cycle — developers can experiment freely, admins can review before deploying to production, rollback if issues found.

Sơ đồ: Plugin Architecture (sandbox + extension points), Extension Points Map (where plugins can hook in), Custom Entity ERD (metadata storage)

Deliverables: Extension Developer Guide, API Documentation for Extensions, Marketplace Specification

Công cụ: WASM sandboxing (Wasmer/Wasmtime), OAuth 2.0 (marketplace auth), custom plugin SDK

Domain 4.3 — Quality Assurance

Chiến lược testing toàn diện — từ unit tests đến performance và security testing.

Module 4.3.1 — Test Strategy & Automation

Mục đích module Bugs trong CRM có business impact trực tiếp: sai pipeline data = forecast sai = hiring decisions sai. Module này build quality safety net toàn diện: automated tests catch regression trước khi reach production, contract tests prevent integration breakage, và shift-left approach phát hiện bugs khi cost-to-fix thấp nhất (coding time vs production hotfix).

DOC-406: CRM Testing Strategy

Mục tiêu: Xây dựng chiến lược QA end-to-end Độ khó: Advanced Critical Thời gian: 10 giờ Tiêu chuẩn: ISTQB, Test Pyramid, Shift-Left

Các bước thực hiện:

  1. Thiết kế Test Pyramid — Define ratio: 70% Unit (domain logic, validators, calculations — fast, isolated), 20% Integration (API endpoints, database queries, service interactions — real dependencies), 10% E2E (critical user journeys — browser automation). Add: Contract Tests (API compatibility between services). Mục đích: fast feedback loop — unit tests run in seconds, catch most bugs at lowest cost. E2E only for highest-value paths.
  2. Implement TDD/BDD Workflow — TDD cho domain logic: Red (write failing test for business rule) → Green (minimum code to pass) → Refactor. BDD cho user stories: Given-When-Then scenarios written with business stakeholders, automated as acceptance tests. Mục đích: tests are specification — written BEFORE code, ensuring implementation matches requirements. No "tests written after to reach coverage target".
  3. Build Test Data Management — Factories (programmatic test data creation — realistic but deterministic), Fixtures (pre-loaded reference data — countries, currencies), Anonymization (production data → safe test data via PII masking). Seed strategy: each test creates its own data, no shared mutable state between tests. Mục đích: reliable, independent tests — test A cannot make test B fail, tests run in any order, parallel execution safe.
  4. Implement Contract Testing — Consumer-Driven Contracts (Pact): frontend defines "I need these fields from API" → contract published → backend CI verifies it still satisfies contract. For events: verify published event schema matches consumer expectations. Mục đích: catch integration breaks at build time — deploying service A doesn't break service B if contracts pass.
  5. Setup Test Environment Strategy — Ephemeral environments: spin up complete CRM stack per PR (containers, test DB, seeded data), run tests, tear down. Shared environments: staging (integration testing, UAT), performance (load testing with production-like data volumes). Mục đích: every PR tested in isolation — no "works on my machine", no "waiting for shared env to be free".

Sơ đồ: Test Pyramid (with percentages and tool per layer), Test Environment Architecture, CI Test Pipeline (stages + gates)

Deliverables: Test Strategy Document, Test Plan Templates (per feature), Coverage Reports (integrated in CI)

Công cụ: Jest/xUnit/JUnit (unit), Playwright (E2E), Pact (contract), Percy (visual regression), axe-core (accessibility)

Module 4.3.2 — Performance & Load Testing

Mục đích module CRM phải responsive với 1000+ concurrent users, sub-second page loads, và handle peak periods (month-end pipeline reviews, marketing campaign launches). Module này establish performance baselines, identify bottlenecks trước production, và ensure system scales gracefully under load thay vì collapse suddenly.

DOC-407: Performance Engineering

Mục tiêu: Đảm bảo CRM xử lý tải production Độ khó: Advanced High Thời gian: 8 giờ Phụ thuộc: DOC-401, DOC-406

Các bước thực hiện:

  1. Define Performance Requirements — SLOs per operation type: page load (<2s P95), API response (<500ms P95), search (<1s P95), report generation (<10s P95). Concurrency targets: 500 simultaneous users (normal), 2000 (peak), 5000 (stress limit). Throughput: 1000 API calls/second sustained. Mục đích: measurable targets — "fast enough" is not a requirement; "P95 < 500ms" is.
  2. Design Load Test Scenarios — Normal load (average business day traffic pattern), Peak load (month-end, all managers reviewing pipeline simultaneously), Stress test (2x peak — find breaking point), Soak test (normal load for 24h — find memory leaks, connection pool exhaustion). Model realistic user behavior: read-heavy (80% reads), mixed operations, concurrent report generation. Mục đích: simulate real usage patterns — not just "hit one endpoint 10K times" but realistic user journeys.
  3. Optimize Database Performance — Identify slow queries (pg_stat_statements, query execution plans). Index strategy: covering indexes for common queries, partial indexes for filtered scans, composite indexes for multi-column filters. Query patterns: avoid N+1 (use eager loading), paginate large result sets, optimize LIKE queries (trigram indexes). Mục đích: database is usually bottleneck #1 — 80% of performance issues are SQL-related, fix them before scaling hardware.
  4. Thiết kế Caching Strategy — L1 cache (in-memory, per-instance — hot config, user sessions), L2 cache (Redis distributed — frequently accessed records, computed values, API responses). Invalidation: event-driven (record updated → cache key evicted), TTL-based (stale data acceptable for N seconds), write-through (cache updated on write). Mục đích: reduce database load 60-80% — most CRM reads are repeat reads (same dashboard, same record list).
  5. Implement APM & Continuous Profiling — Instrument: distributed traces (see full request lifecycle across services), code profiling (CPU + memory hotspots), database query analysis (slow query alerts), frontend performance (Core Web Vitals). Set alerts: P95 degradation >20% from baseline triggers investigation. Mục đích: catch performance regressions in real-time — before users complain, before it becomes P1 incident.

Sơ đồ: Load Test Architecture (test runner → target + monitoring), Caching Layers Diagram (L1/L2 + invalidation flows), Performance Baseline Charts (before/after optimization)

Deliverables: Performance Test Plan (scenarios + acceptance criteria), Benchmark Results (documented baseline), Optimization Recommendations (prioritized by impact)

Công cụ: K6/Gatling (load testing), Grafana (dashboards), New Relic/Datadog (APM), pganalyze (database performance)

Domain 4.4 — DevOps & Deployment

CI/CD pipelines, infrastructure automation và deployment strategy.

Module 4.4.1 — CI/CD & Infrastructure

Mục đích module "Works on my machine" → "Works in production" là khoảng cách lớn nhất trong software. Module này eliminate deployment fear: every commit automatically tested, every deployment reproducible, every environment defined as code. Target: deploy to production multiple times per day with confidence, rollback in < 5 minutes if issues found.

DOC-408: DevOps Pipeline Design

Mục tiêu: Xây dựng CI/CD pipeline cho CRM Độ khó: Advanced Critical Thời gian: 12 giờ Tiêu chuẩn: GitOps, Infrastructure as Code, SRE

Các bước thực hiện:

  1. Choose Git Branching Strategy — Evaluate: Trunk-based (everyone commits to main, feature flags for incomplete work — recommended for experienced teams, fastest feedback), GitFlow (develop/release/hotfix branches — traditional, more ceremony, slower releases), GitHub Flow (feature branches → main — good middle ground). Decision criteria: release cadence, team size, regulatory requirements. Mục đích: entire team agrees on ONE workflow — no ambiguity about where to branch from, how to release, how to hotfix.
  2. Build CI Pipeline — Stages: Compile/Build (fast feedback on syntax errors) → Unit Tests (run in parallel, <3 min) → Integration Tests (containerized dependencies, <10 min) → Security Scan (SAST + dependency audit) → Container Build (Docker image) → Contract Tests → Publish Artifact. Gate: ALL stages must pass before merge allowed. Mục đích: every commit validated automatically — broken code cannot reach main branch, developers get feedback within 15 minutes.
  3. Build CD Pipeline — Stages: Deploy to Dev (auto on merge) → Smoke Tests → Deploy to Staging (auto if smoke pass) → E2E Tests → Manual Gate (for production) → Deploy to Production → Health Checks → Canary Validation (10% → 50% → 100%). Include: automatic rollback if health checks fail post-deploy. Mục đích: deployment is boring, repeatable, and reversible — same process every time, no "special steps for this release".
  4. Implement Infrastructure as Code — Terraform/Pulumi for: cloud resources (VPC, databases, caches, load balancers), Kubernetes clusters, DNS, certificates, IAM roles. GitOps: infrastructure changes go through same PR review as application code. State management: remote backend (S3 + DynamoDB lock), state file per environment. Mục đích: environments are reproducible — disaster recovery = "terraform apply", new environment = clone config + change variables.
  5. Configure Container Orchestration — Kubernetes: define Deployments (replicas, resource limits, health probes), Services (load balancing), Ingress (routing), ConfigMaps/Secrets, HPA (auto-scaling rules). Helm charts or Kustomize for templating across environments. Mục đích: declarative infrastructure — "I want 3 replicas of service X with 2GB RAM max" → Kubernetes makes it so and heals if instance dies.
  6. Implement Secrets Management — HashiCorp Vault or cloud-native (AWS Secrets Manager, Azure Key Vault): store database credentials, API keys, encryption keys. Rotation: automatic rotation on schedule, rotation on suspected compromise. Access: applications pull secrets at runtime (not baked into images), audit log of who accessed what. Mục đích: zero secrets in code, zero secrets in config files, zero secrets in environment variables in CI logs — all dynamic, all audited, all rotatable.

Sơ đồ: CI/CD Pipeline Diagram (all stages + gates + feedback loops), Kubernetes Architecture (per environment), GitOps Flow (PR → merge → deploy)

Deliverables: Pipeline Configuration (as code in repo), IaC Templates (Terraform modules), Runbooks (deployment procedures + troubleshooting)

Công cụ: GitHub Actions/GitLab CI (pipeline), Terraform (IaC), Kubernetes + Helm (orchestration), ArgoCD (GitOps), HashiCorp Vault (secrets)

Module 4.4.2 — Deployment & Release Strategy

Mục đích module Zero-downtime deployment là non-negotiable cho CRM — users span multiple timezones, no acceptable "maintenance window". Module này implement deployment strategies cho phép release bất kỳ lúc nào mà users không nhận ra, plus instant rollback khi issues phát hiện. Bao gồm disaster recovery planning cho worst-case scenarios.

DOC-409: Release Management

Mục tiêu: Quản lý release an toàn và hiệu quả Độ khó: Advanced High Thời gian: 8 giờ Phụ thuộc: DOC-408

Các bước thực hiện:

  1. Chọn Deployment Strategy — Blue-Green (two identical environments, instant switch — best cho stateless services), Canary (gradual rollout 5% → 25% → 100% — best cho detecting issues with subset of traffic), Rolling (replace instances one-by-one — Kubernetes default, good for stateless). For database changes: expand-and-contract pattern (add new column → backfill → migrate reads → remove old column). Mục đích: zero-downtime by design — choose strategy matching your risk tolerance and architecture.
  2. Implement Database Migration Strategy — Rules: migrations must be backward-compatible (old code can still read new schema), separate deploy from migrate (deploy code that handles both schemas, then run migration, then deploy cleanup code). Tools: versioned migrations (Flyway/Liquibase), automated testing of up + down migrations. Mục đích: database is hardest to rollback — if code rollback = 1 minute, schema rollback = hours of data loss risk. Design migrations to be safe in both directions.
  3. Build Rollback Procedures — Automated rollback: health check failures → auto-revert to previous version within 5 minutes. Manual rollback: documented decision criteria (error rate >1%, latency P95 >3x baseline, data corruption detected), execution steps, communication template. Database rollback: pre-tested down migrations, data backup point-in-time recovery. Mục đích: rollback should be one-click, not "emergency all-hands figuring out what to do at 2am".
  4. Implement Feature Flags — Flag types: release flags (hide incomplete feature), operational flags (kill switch for problematic feature), experiment flags (A/B testing), permission flags (beta program access). Lifecycle: flag created → feature developed behind flag → gradual rollout → flag removed (cleanup). Mục đích: decouple deploy from release — deploy code to production daily, enable features for users when ready, disable instantly if problems.
  5. Design Disaster Recovery Plan — RTO (Recovery Time Objective): 1 hour for full system, 15 minutes for critical paths. RPO (Recovery Point Objective): 5 minutes data loss maximum. Strategy: multi-AZ deployment (survive AZ failure), database replication (sync standby), backup verification (restore drill monthly), runbook for each failure scenario (single service, database, full region). Mục đích: when worst case happens (region outage, data corruption), team knows exactly what to do, recovery is practiced not improvised.

Sơ đồ: Deployment Topology (blue-green/canary visualization), Rollback Decision Tree (when + how to rollback), DR Architecture (multi-AZ, failover paths)

Deliverables: Deployment Runbook (step-by-step), DR Plan (tested quarterly), Release Checklist (pre-deploy verification)

Công cụ: Spinnaker/Flux (deployment orchestration), LaunchDarkly/Flagsmith (feature flags), PagerDuty (incident management)

Module 4.4.3 — Data Migration

Mục đích module Data migration là #1 risk factor trong CRM projects — dirty data, lost records, broken relationships can destroy user trust on day 1. Module này build migration pipeline that's tested, validated, and rehearsed multiple times before go-live. Key principle: migration is not a one-time event, it's a repeated, verified process.

DOC-410: Data Migration Strategy

Mục tiêu: Di chuyển dữ liệu từ legacy sang CRM mới Độ khó: Advanced Critical Thời gian: 10 giờ Phụ thuộc: DOC-306, DOC-107

Các bước thực hiện:

  1. Conduct Migration Assessment — Inventory source systems: what data exists, how much (row counts, data size), what format, what quality (completeness, accuracy audit), what relationships (foreign keys, implicit links). Identify: what to migrate (active data), what to archive (historical, read-only access needed), what to discard (obsolete, duplicate). Mục đích: scope the effort accurately — don't discover "oh there's also 2M records in that legacy Access database" at week 3 of migration.
  2. Data Profiling & Cleansing — Profile: value distribution, null percentages, duplicate detection, referential integrity violations, format inconsistencies. Cleansing rules: standardize formats (phone numbers, addresses), resolve duplicates (merge strategy with master record selection), fill gaps (default values or flag for manual review). Mục đích: "garbage in, garbage out" — clean BEFORE migrating, not after (post-migration cleanup is 10x harder because users already see dirty data).
  3. Build Field Mapping & Transformation — For each source field → target field: direct mapping (1:1), transformation (format conversion, concatenation, split), derivation (computed from multiple source fields), enrichment (add data from external source), default (no source equivalent). Document business rules per mapping. Mục đích: explicit, testable mapping — no ambiguity about "how does legacy status code 'A3' map to new system's 'Qualified' stage?"
  4. Design Migration Pipeline — Architecture: Extract (connectors to each source system) → Transform (apply mapping rules, cleanse, validate) → Load (write to target CRM via API or direct DB). Include: error handling (log failures, continue processing, retry logic), checkpoint/restart (resume from last successful batch), audit trail (track every record: source → target mapping). Mục đích: repeatable, automated pipeline — run it 10 times in testing before running once in production.
  5. Implement Validation & Reconciliation — Post-migration checks: record count matching (source vs target), data integrity (referential integrity in target), business rule validation (all required fields populated, status values valid), sample verification (random 100 records manually spot-checked). Automated reconciliation report. Mục đích: prove migration is correct BEFORE go-live — "we migrated 1.2M accounts, reconciliation shows 99.97% match, 360 records in exception queue for manual review".
  6. Plan Cutover & Rollback — Cutover strategy: incremental (migrate nightly, final delta sync on go-live day) vs big-bang (one-time full migration during maintenance window). Cutover timeline: T-24h freeze source edits → T-12h start migration → T-6h validation → T-2h stakeholder sign-off → T-0 switch over. Rollback: maintain source system access for 30 days, data export from new system if needed. Mục đích: go-live is choreographed — everyone knows their role, timing, and decision criteria for go/no-go.

Sơ đồ: Migration Pipeline Architecture, Field Mapping Matrix, Cutover Timeline

Deliverables: Migration Plan, Mapping Document, Validation Report, Cutover Checklist

Công cụ: Talend, Informatica, Apache NiFi, custom ETL scripts, dbt
