Domain 6.1 — AI & Machine Learning for CRM

Tích hợp AI/ML để biến CRM từ record-keeping system thành intelligent decision-support platform.

Module 6.1.1 — Predictive Analytics & ML Models

Mục đích module AI biến CRM từ "system of record" (ghi chép quá khứ) thành "system of intelligence" (dự đoán tương lai + recommend actions). Module này xây dựng ML pipeline cho 4 use cases có ROI cao nhất: Lead Scoring (tập trung sales vào deals likely-to-close), Churn Prediction (intervene trước khi khách rời), CLV (invest đúng khách hàng), Next-Best-Action (guide reps on what to do next).

DOC-601: AI-Powered CRM Intelligence

Mục tiêu: Xây dựng AI layer cho CRM platform Độ khó: Expert Critical Thời gian: 12 giờ Tiêu chuẩn: MLOps, Responsible AI, EU AI Act

Các bước thực hiện:

  1. Identify & Prioritize ML Use Cases — Evaluate CRM-specific ML opportunities: Lead Scoring (predict conversion probability), Churn Prediction (predict customers at risk), CLV Prediction (forecast future value), Next-Best-Action (recommend what rep should do), Sentiment Analysis (gauge customer mood from interactions). Score each: business impact × data availability × feasibility. Start with highest-scored. Mục đích: avoid "AI for AI's sake" — each model must have measurable business outcome and sufficient training data to be viable.
  2. Design Feature Engineering from CRM Data — Extract ML features from: behavioral signals (email opens, page views, login frequency), transactional data (purchase history, deal size trend), temporal patterns (days since last contact, response time trend), relationship signals (stakeholder count, org chart coverage), firmographic data (company size, industry, funding stage). Build Feature Store for reuse across models. Mục đích: features are the "raw material" of ML — good features matter more than algorithm choice. Feature Store enables consistency between training and serving.
  3. Build MLOps Pipeline — End-to-end: Data Collection → Feature Engineering → Model Training (versioned experiments) → Evaluation (offline metrics: AUC, precision, recall) → Deployment (A/B canary rollout) → Monitoring (model drift detection, performance degradation alerts) → Retraining (scheduled or triggered by drift). Mục đích: reproducible, automated ML lifecycle — not "data scientist trains in notebook, throws pickle over wall to engineering".
  4. Thiết kế Model Serving Architecture — Real-time (lead scoring at point of form submission — <100ms latency requirement): model served via API, pre-computed features from Feature Store. Batch (CLV recalculation — nightly): scheduled job processes all customers, writes scores to CRM database. Near-real-time (churn risk update — event-triggered): streaming pipeline recalculates on key events (support ticket created, usage dropped). Mục đích: match serving pattern to use case SLA — not everything needs real-time; batch is simpler and cheaper when acceptable.
  5. Implement Explainability & Responsible AI — For each model: SHAP values (which features drove this prediction?), Model Cards (document performance across demographic groups), Bias Audit (does model unfairly disadvantage any customer segment?), Human Override (user can disagree with AI recommendation). Comply with EU AI Act transparency requirements. Mục đích: trust requires explainability — sales rep won't follow "AI score = 85" without understanding why. Bias detection prevents discriminatory outcomes.
  6. Design A/B Testing for ML Models — Compare: model predictions vs human decisions (does ML improve outcomes?), new model vs existing model (does V2 beat V1?), model-assisted vs control group (does providing AI score improve rep behavior?). Track: conversion rate lift, revenue impact, time saved. Statistical rigor: proper sample size, control for confounders. Mục đích: prove AI value with data — "model accuracy improved 5%" means nothing if it doesn't translate to business outcome improvement.

Sơ đồ: ML Pipeline Architecture (end-to-end flow), Feature Store Design (shared features across models), Model Serving Topology (real-time vs batch paths)

Deliverables: ML Strategy Document (use cases + priority + timeline), Model Cards (per model: purpose, performance, limitations, bias assessment), MLOps Pipeline Specification

Công cụ: MLflow (experiment tracking), Kubeflow/SageMaker/Vertex AI (pipeline orchestration), Feast/Tecton (Feature Store), SHAP (explainability)

Module 6.1.2 — NLP & Conversational AI

Mục đích module LLMs đã thay đổi game cho CRM: auto-draft emails, summarize call notes, extract action items, answer questions from knowledge base, và assist agents in real-time. Module này integrate generative AI safely: maximize productivity gains while preventing hallucination, data leaks, và inappropriate outputs. Key: AI assists humans, không replace human judgment on critical decisions.

DOC-602: Conversational AI for CRM

Mục tiêu: Tích hợp NLP và conversational AI Độ khó: Expert High Thời gian: 10 giờ Phụ thuộc: DOC-601, DOC-208

Các bước thực hiện:

  1. Identify LLM Integration Points — High-value CRM use cases: Email Drafting (given context + tone, generate reply), Meeting Summarization (transcribe + extract action items + update CRM fields), Data Entry Assist (extract contact info from email signatures), Record Summarization ("tell me about this account in 30 seconds"), Search Enhancement (natural language → structured query). Prioritize by: time saved × frequency × accuracy achievable. Mục đích: target repetitive, low-judgment tasks first — max productivity gain with lowest hallucination risk.
  2. Build RAG Architecture for Knowledge Retrieval — Pipeline: CRM knowledge base + product docs + internal wiki → chunked + embedded → vector store. At query time: user question → embed → similarity search → top-K chunks → LLM generates answer grounded in retrieved context. Include: source attribution (show which documents informed the answer), confidence scoring, fallback to "I don't know". Mục đích: chatbot answers questions accurately from internal knowledge rather than hallucinating — critical for customer-facing use where wrong answers = trust damage.
  3. Implement Voice AI Integration — Call transcription: real-time (Whisper/Deepgram) → transcript stored with recording. Post-call processing: extract next steps, identify objections raised, detect sentiment shifts, score against sales methodology (did rep follow MEDDPICC?). Real-time coaching: whisper suggestions to agent during live call (competitor mentioned → show battle card). Mục đích: unlock insights from 100% of calls (not just the 5% managers can listen to), and provide real-time assistance that improves every conversation.
  4. Design Email Intelligence — Auto-categorize inbound emails (support request, sales inquiry, renewal discussion, feedback). Priority scoring (urgency indicators, VIP customer, revenue at risk). Suggested responses (template selection + personalization). Thread summarization (collapse 20-email thread into 3 bullet points). Mục đích: reduce time-in-inbox for reps and agents — auto-routing + suggested responses = faster handling, less cognitive load.
  5. Implement AI Safety & Guardrails — Protections: input sanitization (prompt injection prevention), output validation (no PII leakage, no hallucinated data, no inappropriate content), human-in-the-loop (AI drafts, human approves before send), audit logging (all AI inputs/outputs recorded), rate limiting (prevent abuse), model output filtering (toxicity, bias). Compliance: data residency (customer data stays in region), consent (clear disclosure when AI is involved). Mục đích: AI that's powerful AND safe — one leaked customer record or offensive auto-response = trust destruction and regulatory exposure.

Sơ đồ: Conversational AI Architecture (LLM + RAG + guardrails), RAG Pipeline (embed → store → retrieve → generate), Intent Classification Flow (chatbot decision tree)

Deliverables: AI Integration Specification, Prompt Library (tested + versioned prompts per use case), AI Safety Policy Document (guardrails + acceptable use)

Công cụ: OpenAI/Anthropic APIs (LLM), LangChain/LlamaIndex (orchestration), Pinecone/Weaviate (vector DB), Whisper/Deepgram (speech-to-text), Guardrails AI (output validation)

Module 6.1.3 — Recommendation & Personalization Engine

Mục đích module "Next-Best-Action" là killer feature cho CRM — thay vì rep tự quyết "nên gọi ai hôm nay?", hệ thống recommend dựa trên data: "Contact John at Acme (engagement spike yesterday, proposal viewed 3x, competitor meeting next week → urgency HIGH)". Module này build personalization engine cho both internal users (rep guidance) AND external interactions (customer experience personalization).

DOC-603: Intelligent Recommendations

Mục tiêu: Xây dựng recommendation engine cho CRM Độ khó: Expert High Thời gian: 8 giờ Phụ thuộc: DOC-601, DOC-210

Các bước thực hiện:

  1. Design Next-Best-Action Engine — For sales reps: rank daily priorities (which accounts to contact, which deals need attention, what actions to take). Factors: engagement recency, deal stage momentum, buyer signals, competitive threats, relationship strength decay. Output: prioritized action list with reason ("Contact: engagement spike + proposal viewed"). Mục đích: reps spend first hour on highest-impact activities instead of inbox triaging — data-driven daily prioritization beats intuition.
  2. Build Content Recommendation — For marketing/sales: given customer profile + current journey stage → recommend: which content to share (case study matching their industry), which email template to use, which product to cross-sell, which event to invite to. Algorithms: collaborative filtering (customers like them engaged with X), content-based (profile matches content tags). Mục đích: right content at right time — instead of "blast same whitepaper to everyone", personalize based on where they are in their journey.
  3. Implement Real-time Personalization — For customer-facing experiences: website personalization (show relevant case studies, pricing page variant), email personalization (dynamic content blocks based on segment), chatbot personalization (greeting + context + suggestions based on profile). Event-driven: profile update → immediate impact on next interaction. Mục đích: every customer interaction feels "they know me" — personalization increases conversion 20-40% vs generic.
  4. Design Dynamic Pricing & Offer Optimization — For sales/renewals: recommend optimal discount level (balance win probability vs margin), suggest best bundle/package (maximize customer value + revenue), predict price sensitivity per segment. Guardrails: minimum margin thresholds, approval required above X% discount, fairness constraints (no discriminatory pricing). Mục đích: maximize revenue per deal while maintaining fair pricing — AI suggests, human approves, compliance enforces.
  5. Implement Exploration vs Exploitation (Contextual Bandits) — Problem: pure exploitation (always recommend what worked before) misses opportunities. Solution: contextual bandits allocate small % of recommendations to explore (try new approaches) while primarily exploiting (use proven winners). Track: regret metric, exploration rate over time, discovery of new high-performing approaches. Mục đích: recommendation system continuously improves itself — discovers new effective patterns without waiting for A/B test to be manually designed.

Sơ đồ: Recommendation Architecture (data → features → model → ranking → delivery), Personalization Pipeline (profile → segment → rule evaluation → content selection), Decision Flow (exploration vs exploitation)

Deliverables: Recommendation Engine Specification, Evaluation Metrics (precision, recall, business impact), A/B Test Framework (for validating recommendation quality)

Công cụ: Apache Spark MLlib (collaborative filtering), TensorFlow Recommenders, Amazon Personalize (managed service), custom contextual bandit implementation

Domain 6.2 — Platform Engineering

Xây dựng CRM thành true platform — multi-tenant SaaS, marketplace, và developer ecosystem.

Module 6.2.1 — Multi-Tenant SaaS Architecture

Mục đích module Nếu CRM sẽ serve multiple customers (SaaS model), multi-tenancy là foundational architecture decision. Module này thiết kế tenant isolation đảm bảo: data security (tenant A never sees tenant B), performance isolation (one tenant's heavy usage doesn't degrade others), và customization flexibility (each tenant configures differently without code forks). Wrong tenancy model = rewrite later.

DOC-604: SaaS Platform Design

Mục tiêu: Thiết kế multi-tenant SaaS CRM Độ khó: Expert Critical Thời gian: 12 giờ Tiêu chuẩn: SaaS Maturity Model, AWS Well-Architected SaaS Lens

Các bước thực hiện:

  1. Chọn Tenancy Model — Evaluate 3 models: Silo (separate infrastructure per tenant — max isolation, highest cost, simplest logic), Pool (shared everything with logical separation — lowest cost, most efficient, hardest to guarantee isolation), Bridge (hybrid — shared compute, separate databases per tenant tier). Decision criteria: compliance requirements, data sensitivity, cost targets, number of tenants, customization needs. Mục đích: foundational decision affecting all subsequent architecture — choose wrong = massive refactoring later.
  2. Thiết kế Tenant Isolation — Data isolation: Row-Level Security (RLS) with tenant_id in every query, separate schemas, or separate databases. Compute isolation: shared pods with resource quotas (Kubernetes), separate node pools for premium tenants. Network isolation: network policies, separate VPCs for high-security tenants. Test: prove data leak impossible via automated isolation tests. Mục đích: existential for SaaS — one tenant data leak = company-ending event.
  3. Build Onboarding Automation — Self-service signup: create tenant record → provision resources → seed default data → configure defaults → welcome email with setup wizard. Under 5 minutes from signup to first login. Support: trial → paid conversion (upgrade without disruption), custom onboarding for enterprise (white-glove with data import). Mục đích: time-to-value determines conversion — every minute of onboarding friction loses a percentage of signups.
  4. Implement Billing & Metering — Track usage per tenant: API calls, storage, users, features accessed. Pricing models: per-seat (simple), usage-based (fair), tiered (good-better-best), hybrid. Integration: CRM events → metering service → billing platform (Stripe) → invoice. Handle: upgrades/downgrades mid-cycle, proration, overages. Mục đích: revenue engine of SaaS — accurate metering = fair billing = customer trust = reduced churn from billing disputes.
  5. Design Noisy Neighbor Prevention — Detect: one tenant consuming disproportionate resources (CPU, database connections, API calls). Prevent: per-tenant rate limits, per-tenant resource quotas, per-tenant queue priorities, throttling heavy operations (bulk imports, report generation). Alert: tenant approaching limits → notification before hard throttle. Mục đích: one enterprise tenant running huge report shouldn't make platform slow for 1000 SMB tenants — isolation at performance level, not just data level.

Sơ đồ: Multi-tenant Architecture (showing isolation boundaries), Tenant Isolation Diagram (data + compute + network layers), Onboarding Flow (signup → provisioning → active)

Deliverables: SaaS Architecture Document, Tenant Isolation Specification (with proof of isolation), Pricing Model Document

Công cụ: AWS SaaS Factory (reference architecture), Kubernetes namespaces + resource quotas, Stripe Billing (metering + invoicing), LaunchDarkly (per-tenant feature flags)

Module 6.2.2 — Marketplace & Developer Ecosystem

Mục đích module Platform = product + ecosystem. Salesforce's AppExchange has 7000+ apps — that's platform power. Module này build marketplace cho third-party developers to extend CRM: add integrations, add features, serve vertical-specific needs that the core platform can't cover. Key insight: you can't build everything yourself — but you can build a platform that lets others build on top.

DOC-605: CRM Marketplace Platform

Mục tiêu: Xây dựng app marketplace cho CRM Độ khó: Expert High Thời gian: 8 giờ Phụ thuộc: DOC-604, DOC-405

Các bước thực hiện:

  1. Design Marketplace Architecture — Components: App Registry (metadata, versions, pricing), Installation Engine (per-tenant app provisioning), Runtime Sandbox (isolated execution), Review Pipeline (automated + manual security review), Billing Integration (revenue share, customer billing). App types: integrations (connect external systems), add-ons (add features to UI), templates (pre-built workflows/reports). Mục đích: platform for scale — ecosystem of partner-built apps extends CRM value without your engineering team bottleneck.
  2. Build Developer Portal & SDK — Portal contents: getting started guide (<30 min to "hello world"), API reference (interactive), SDK libraries (JavaScript, Python, .NET), sample apps (common patterns), sandbox environment (test without affecting production), community forum. SDKs abstract: authentication, API calls, event subscriptions, UI components. Mục đích: developer experience determines ecosystem health — if building an app takes weeks instead of days, partners won't invest.
  3. Implement App Security Review — Automated: SAST scan (code vulnerabilities), dependency audit (known CVEs), permission scope review (does it request more access than needed?), performance testing (resource usage within limits). Manual: UX review, functionality testing, data handling review. Certification levels: basic (automated only), verified (manual review), premium (ongoing monitoring). Mục đích: customer trust requires app quality — one malicious/buggy app damages platform reputation.
  4. Design Revenue Sharing Model — Standard split: 70% developer / 30% platform (industry norm). Tiers: higher developer share for verified/premium apps. Payment: monthly payouts, minimum threshold, transparent reporting. Additional incentives: co-marketing for top apps, featured placement, early access to new APIs. Mục đích: attract quality developers — revenue share must be compelling enough to justify their investment in building on your platform vs competitors.
  5. Build Partner Certification Program — Tiers: Registered (basic access), Certified (trained + tested), Premier (deep expertise + co-sell). Requirements per tier: training completion, technical assessment, customer references, revenue threshold. Benefits: lead referrals, co-marketing, dedicated support, early API access. Mục đích: quality control on professional services ecosystem — customers need confidence that "certified partner" means competent implementation.

Sơ đồ: Marketplace Architecture (app lifecycle), App Installation Flow (customer → marketplace → install → sandbox), Partner Ecosystem Map (tiers + relationships)

Deliverables: Marketplace Specification, Developer Guide (comprehensive), Partner Program Document (tiers + benefits + requirements)

Công cụ: ReadMe/Stoplight (developer portal), OAuth 2.0 (app authorization), WASM/container sandboxing (app isolation), Stripe Connect (revenue sharing)

Module 6.2.3 — Low-Code/No-Code Platform

Mục đích module IT bottleneck is real: business users have ideas faster than engineering can build. Low-code platform empowers "citizen developers" (admins, power users) to build custom apps, forms, workflows, and reports WITHOUT writing code. Module này build visual builder tools that are powerful enough for real use cases yet safe enough (governance, guardrails) to not break production.

DOC-606: Low-Code CRM Builder

Mục tiêu: Cho phép citizen developers tùy biến CRM Độ khó: Expert Medium Thời gian: 8 giờ Phụ thuộc: DOC-404, DOC-405

Các bước thực hiện:

  1. Build Visual Form Designer — Drag-drop interface: field types (text, number, date, lookup, file upload, signature), layout (sections, columns, tabs), logic (show/hide rules, required conditions, validation), submission actions (create record, send notification, trigger workflow). Preview mode: test form without publishing. Mục đích: admin creates custom data capture forms in minutes — onboarding questionnaire, feedback survey, expense request, custom intake form.
  2. Implement Visual Workflow Builder — Complement to DOC-404 (developer workflow engine) with visual no-code interface: trigger selection (point-and-click), condition builder (field + operator + value, no code), action library (send email, create task, update field, call API), testing mode (simulate with sample data). Mục đích: business users automate own processes — "when deal > $50K, notify VP" shouldn't need a developer ticket.
  3. Build Custom Report/Dashboard Builder — Self-service: select data source (entity), add filters (visual), choose visualization (table, chart, KPI card), configure drill-down, schedule email delivery. Support: calculated fields (formula builder), cross-object reporting (joins without SQL), conditional formatting (highlight if metric > threshold). Mục đích: eliminate "please build me a report" requests — business users explore data independently, reducing analytics team backlog by 60%+.
  4. Implement Governance & Guardrails — Prevent citizen-dev chaos: role-based builder access (not everyone can modify production), approval workflow for publishing (admin review before live), version history (rollback any change), impact analysis (show "this change affects 500 users"), resource limits (prevent infinite loops in workflows, query timeouts in reports). Mục đích: empowerment WITH safety — one bad workflow trigger can spam 10,000 emails; governance prevents disasters while maintaining speed.
  5. Design Version Control for Citizen Dev — Track: who changed what, when, why (optional comment). Support: draft mode (work without affecting live), publish (make live for users), rollback (instant revert to previous version), compare (diff between versions). Audit trail: all changes logged for compliance. Mục đích: enterprise requirement — know who changed the lead routing rule last Tuesday and be able to undo it in one click.

Sơ đồ: Low-code Platform Architecture (builder → metadata → runtime), Builder UI Wireframes (form designer + workflow builder), Governance Model (roles + approval + audit)

Deliverables: Low-code Platform Specification, Builder User Guide (for admins), Governance Policy Document (what's allowed, review process)

Công cụ: Retool/Appsmith (reference for internal tools), custom visual builder framework, metadata-driven rendering engine

Domain 6.3 — Emerging Technologies

Công nghệ mới nổi sẽ reshape CRM — IoT, blockchain, AR/VR và composable architecture.

Module 6.3.1 — IoT & Connected Customer Experience

Mục đích module IoT biến CRM từ reactive ("customer calls to report problem") thành proactive ("system detects issue before customer notices"). Applicable cho: connected devices (smart home, industrial equipment, vehicles), wearables, retail sensors. Module này integrate IoT telemetry vào CRM: device health → predictive service tickets, location data → contextual marketing, usage patterns → product recommendations.

DOC-607: IoT-Connected CRM

Mục tiêu: Tích hợp IoT signals vào CRM Độ khó: Expert Medium Thời gian: 6 giờ Phụ thuộc: DOC-307, DOC-601

Các bước thực hiện:

  1. Design IoT Data Ingestion Pipeline — Architecture: devices → IoT Gateway (MQTT/HTTP) → stream processing (filter, aggregate, enrich) → CRM events (only business-relevant signals forwarded). Volume management: raw telemetry stays in time-series DB, only actionable events reach CRM. Example: "device temperature > threshold for 10 minutes" → create service case automatically. Mục đích: CRM doesn't drown in raw sensor data — only business-meaningful events arrive, processed and enriched with customer context.
  2. Implement Proactive Service Triggers — ML models on device telemetry predict failures before they happen: "this motor's vibration pattern matches failure pattern — 85% probability of failure within 7 days". Auto-action: create service case, schedule preventive maintenance, notify customer, order replacement parts. Mục đích: shift from "break-fix" to "predict-prevent" — customer never experiences downtime, service costs drop (planned maintenance vs emergency dispatch).
  3. Build Location-based CRM Features — Geofencing: customer enters store zone → trigger welcome notification + load preferences for staff. Proximity: field service technician near customer location → suggest opportunistic visit. Territory: real-time map of customer locations + rep territories + visit history. Mục đích: physical-world context enriches digital CRM — bridge online-offline customer experience.
  4. Design Digital Twin Integration — For B2B/industrial: create virtual representation of customer's physical asset in CRM. Twin shows: current state, maintenance history, predicted lifecycle, optimal usage recommendations. Link: service cases → affected twin, sales opportunities → expansion based on twin insights. Mục đích: sales and service teams "see" the customer's reality without site visit — informed conversations, proactive recommendations.

Sơ đồ: IoT-CRM Integration Architecture (devices → gateway → processing → CRM), Event Stream Processing Pipeline, Digital Twin Model (asset → twin → CRM entities)

Deliverables: IoT Integration Specification, Use Case Library (industry-specific), Data Pipeline Design (ingestion + processing + routing)

Công cụ: AWS IoT Core/Azure IoT Hub (gateway), Apache Kafka/Flink (stream processing), TimescaleDB (time-series storage), custom ML models (predictive maintenance)

Module 6.3.2 — Blockchain & Decentralized Identity

Mục đích module Blockchain cho CRM không phải về cryptocurrency — mà về trust, verifiable identity, và customer data ownership. Module này explore: customer controls their own data (self-sovereign), loyalty points as transferable tokens, và smart contracts automating service agreements. Applicable khi: trust between parties is low, data portability is required, or tokenized incentives make business sense.

DOC-608: Trust & Decentralized CRM

Mục tiêu: Áp dụng blockchain cho trust và identity Độ khó: Expert Medium Thời gian: 4 giờ Phụ thuộc: DOC-308

Các bước thực hiện:

  1. Assess Blockchain Use Cases for CRM — Evaluate viability: Decentralized Identity (DIDs — customer proves identity without sharing raw PII), Loyalty Tokenization (points as transferable tokens on chain — interoperable between partners), Smart Contract SLAs (auto-execute penalties/credits when SLA breached), Supply Chain Provenance (product authenticity verification linked to customer record). Score each: business value × technical feasibility × regulatory acceptance. Mục đích: filter hype from value — most blockchain CRM ideas don't justify the complexity. Identify the 1-2 that genuinely solve trust problems not solvable otherwise.
  2. Design Decentralized Identity Integration — Architecture: customer creates DID (decentralized identifier) + wallet → issues Verifiable Credentials (VCs) from trusted parties (bank KYC, employer verification) → presents VCs to CRM for verification without revealing underlying data. CRM stores: proof of verification, not raw PII. Benefits: reduced PII liability, customer portability, fraud reduction. Mục đích: shift liability — instead of storing 1M customers' passport numbers, store cryptographic proof they were verified. Data breach impact dramatically reduced.
  3. Design Tokenized Loyalty Program — Architecture: loyalty points as tokens on blockchain (fungible — ERC-20 equivalent). Features: earn from purchases/engagement, spend across partner network, trade/gift to others, expire via smart contract logic. Benefits vs traditional: interoperability (partner ecosystem without bilateral agreements), transparency (customer sees all earn/burn on-chain), reduced fraud (tamper-proof balances). Mục đích: next-gen loyalty that's portable, transparent, and composable — differentiator for customer acquisition and retention.
  4. Implement Privacy-Preserving Data Sharing — Zero-knowledge proofs: prove attributes without revealing data ("customer is over 18" without revealing birthdate, "customer spent >$10K this year" without revealing exact transactions). Use for: partner data sharing (share insights without raw data), compliance verification (prove consent without exposing consent records), analytics on encrypted data. Mục đích: unlock data value while maintaining privacy — enable use cases that GDPR would otherwise block.

Sơ đồ: DID Architecture (wallet → issuer → verifier → CRM), Token Economy Design (earn/burn/transfer flows), Smart Contract Flow (SLA monitoring → breach detection → auto-credit)

Deliverables: Blockchain Use Case Assessment (scored viability matrix), DID Integration Specification (if applicable)

Công cụ: Hyperledger (enterprise permissioned), Ethereum/Polygon (public + L2), Ceramic Network (decentralized data), SpruceID (DID toolkit)

Module 6.3.3 — Composable CRM & MACH Architecture

Mục đích module Composable architecture is the future of enterprise software — instead of monolithic suite (Salesforce does everything), compose best-of-breed components via APIs. MACH principles (Microservices, API-first, Cloud-native, Headless) enable: swap any component without rewriting the rest, choose best tool per capability, and avoid vendor lock-in. Module này design CRM as composable platform — both consuming external PBCs AND being consumed as a PBC by others.

DOC-609: Composable CRM Architecture

Mục tiêu: Thiết kế CRM theo MACH principles Độ khó: Expert High Thời gian: 8 giờ Tiêu chuẩn: MACH Alliance, Packaged Business Capabilities

Các bước thực hiện:

  1. Apply MACH Principles to CRM — Evaluate current architecture against 4 principles: Microservices (each capability independently deployable?), API-first (all functionality accessible via API, not just UI?), Cloud-native (designed for cloud elasticity, not lift-and-shift?), Headless (backend decoupled from any specific frontend?). Gap analysis: which principles are met, which need architectural changes? Mục đích: assess composability readiness — know exactly where you are vs where MACH target state requires.
  2. Design Packaged Business Capabilities (PBCs) — Decompose CRM into PBCs: each PBC is self-contained, API-accessible, independently replaceable. Examples: "Lead Management PBC", "Email Campaign PBC", "Case Resolution PBC", "Customer 360 PBC". Each PBC has: defined API contract, event publications, data ownership, SLA. Mục đích: any PBC can be swapped for best-of-breed alternative — if better email tool emerges, replace Email Campaign PBC without touching Lead Management.
  3. Build Orchestration Layer — Orchestration sits between PBCs: routes requests, coordinates cross-PBC workflows (saga patterns), handles event routing, manages API composition (aggregate data from multiple PBCs for single UI request). Implementation: API Gateway + Event Mesh + BFF layer. Mục đích: PBCs don't know about each other directly — orchestration provides the "glue" that makes independent components work together as coherent system.
  4. Design Best-of-Breed Integration Strategy — For each capability: evaluate build-in-house vs adopt external PBC. Decision criteria: strategic differentiation (build), commodity (buy best-of-breed), data sensitivity (keep in-house), integration complexity (prefer fewer vendors). Create technology radar: Adopt (proven, use now), Trial (promising, limited use), Assess (interesting, evaluate), Hold (avoid). Mục đích: pragmatic composition — not everything needs to be built or bought. Right mix based on strategic importance and team capacity.
  5. Implement Vendor Lock-in Mitigation — Strategies: abstraction layers (don't depend directly on vendor SDKs — wrap with internal interface), data portability (export all data in standard formats at any time), contract flexibility (avoid multi-year lock-in, negotiate exit clauses), standards compliance (OpenAPI, AsyncAPI, CloudEvents — vendor-neutral contracts). Test: "how long to replace vendor X?" should be weeks, not months. Mục đích: composable = freedom to change. If replacing a component takes 6 months of rewriting, you're not composable — you're locked in with extra steps.
