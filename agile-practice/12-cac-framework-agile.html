Tổng Quan Frameworks Agile

Scrum, XP, Kanban, Crystal, FDD, DSDM — so sánh và lựa chọn phù hợp

Nguồn: Agile Practice Guide — Annex A3: Overview of Agile and Lean Frameworks
Tư duy cốt lõi Không có "best framework" cho mọi tình huống. Mỗi framework có strengths riêng. Đội thực tiễn thường pha trộn elements từ nhiều frameworks để phù hợp bối cảnh cụ thể.

Scrum

Framework phổ biến nhất, process-framework nhẹ cho phát triển và duy trì sản phẩm phức tạp.

3 Vai trò

  • Product Owner — sở hữu backlog, ưu tiên theo value, đại diện stakeholders
  • Scrum Master — servant leader, gỡ impediments, coaching Scrum
  • Development Team — cross-functional, self-organizing, 3-9 người

5 Events

EventMục đíchTimebox
SprintContainer cho tất cả events, giao increment1-4 tuần
Sprint PlanningChọn items từ backlog, tạo sprint plan≤8 giờ (sprint 4 tuần)
Daily ScrumSynchronize, identify impediments≤15 phút
Sprint ReviewDemo increment, thu feedback≤4 giờ
Sprint RetrospectiveInspect & adapt process≤3 giờ

3 Artifacts

  • Product Backlog — ordered list of everything needed in product
  • Sprint Backlog — items selected cho sprint + plan để deliver
  • Increment — sum of all completed items, phải usable

eXtreme Programming (XP)

Nhấn mạnh engineering practices và technical excellence. Tên gọi "eXtreme" vì đẩy best practices lên mức cực đoan.

Core Practices

PracticeMô tả
Pair Programming2 developers cùng 1 máy — continuous review, knowledge sharing
Test-Driven DevelopmentWrite test first → code → refactor
Continuous IntegrationIntegrate code nhiều lần/ngày, automated build & test
RefactoringCải thiện code structure liên tục mà không thay đổi behavior
Simple DesignSimplest thing that works — YAGNI
Collective OwnershipMọi người có thể change bất kỳ code nào
Coding StandardsThống nhất code style để collective ownership work
Sustainable Pace40-hour weeks — no overtime as norm
On-site CustomerReal customer available cho đội hàng ngày
Small ReleasesRelease thường xuyên, increments nhỏ
XP + Scrum Scrum cho process framework (events, roles), XP cho engineering practices (TDD, CI, pair programming). Kết hợp rất phổ biến và hiệu quả.

Kanban Method

Phương pháp evolutionary change management — bắt đầu từ hiện tại, cải tiến dần (start-where-you-are).

6 Practices

  1. Visualize the workflow — Kanban board với columns cho mỗi state
  2. Limit Work in Progress — WIP limits cho mỗi column
  3. Manage flow — Monitor và optimize flow of work items
  4. Make policies explicit — Rules rõ ràng cho mỗi transition
  5. Implement feedback loops — Cadences cho review và improvement
  6. Improve collaboratively, evolve experimentally — Dùng models và experiments

4 Foundational Principles

  • Start with what you do now
  • Agree to pursue incremental, evolutionary change
  • Respect the current process, roles, responsibilities, and titles
  • Encourage acts of leadership at all levels

Kanban vs. Scrum

AspectScrumKanban
CadenceFixed sprints (1-4 tuần)Continuous flow
RolesPO, SM, Dev TeamKhông prescribed roles
ChangeKhông thay đổi trong sprintCó thể thêm anytime (if WIP allows)
EstimationStory points / velocityLead time / throughput
PrescriptiveModerateMinimal

Crystal Family

Họ các phương pháp (Clear, Yellow, Orange, Red...) được scale theo team size và criticality. Tạo bởi Alistair Cockburn.

Core Properties

  • Frequent delivery — Giao code chạy được cho users thường xuyên
  • Reflective improvement — Đội tự suy ngẫm và cải thiện
  • Osmotic communication — Thông tin flows tự nhiên khi đội ngồi gần
  • Personal safety — An toàn để nói thật mà không sợ hậu quả
  • Focus — Biết rõ 2 giờ tiếp theo sẽ làm gì
  • Easy access to expert users — Users available cho đội
  • Technical environment with automated tests, CI

Đặc điểm

  • Nhẹ nhất trong các methodologies — ít ceremonies nhất
  • Scale theo "color" — Clear (≤8 người), Yellow (≤20), Orange (≤50), Red (≤100)
  • Mỗi color thêm structure/ceremonies theo nhu cầu quy mô

Feature-Driven Development (FDD)

Agile method tổ chức quanh features. Phù hợp đội lớn hơn và dự án có domain complexity cao.

5 Processes

Develop Model → Build Feature List → Plan by Feature → Design by Feature → Build by Feature
  • Iterations ngắn (2 tuần) tập trung vào features
  • Domain modeling upfront rồi iterative delivery
  • Mỗi feature đủ nhỏ để deliver trong 2 tuần
  • Reporting dựa trên features completed

DSDM (Dynamic Systems Development Method)

Approach cố định thời gian và chi phí, thay đổi scope. Nguyên tắc chính: thời gian và chi phí cố định, scope linh hoạt (MoSCoW prioritization).

MoSCoW Prioritization

  • Must have — Bắt buộc, không có thì delivery thất bại
  • Should have — Quan trọng nhưng không critical cho delivery
  • Could have — Desired nhưng có thể bỏ nếu hết thời gian
  • Won't have (this time) — Sẽ không làm trong iteration này

Đặc điểm DSDM

  • Focus vào business need và value
  • Deliver on time bằng cách flex scope
  • Collaborate với stakeholders liên tục
  • Never compromise quality
  • Build incrementally from firm foundations

Lựa Chọn Framework

Câu hỏi quyết định

  1. Đội có experience với Agile chưa? → Mới = Scrum (structured) hoặc Kanban (gentle); kinh nghiệm = XP hoặc blend
  2. Cần engineering excellence? → XP practices (TDD, pair, CI)
  3. Đã có process, muốn improve dần? → Kanban (start-where-you-are)
  4. Fixed deadline, flexible scope? → DSDM + MoSCoW
  5. Large team, complex domain? → FDD hoặc Scrum + scaling
  6. Small team, high trust? → Crystal Clear (minimal overhead)
Thực tế phổ biến nhất Hầu hết đội thành công pha trộn: Scrum framework (roles, events) + Kanban board (visualization, WIP) + XP practices (TDD, CI, refactoring). Đừng chọn 1, hãy craft approach phù hợp — nhưng hiểu rõ từng framework trước khi pha trộn.
Agile frameworks không được customized cho đội. Đội có thể cần tailor practices để deliver value thường xuyên. Thường các đội practice blend riêng của mình, dù dùng một framework cụ thể làm starting point.
